# Frontend MVP

桥梁病害识别系统的前端当前已经完成 Phase 2 闭环，并在完成 Phase 3 结果展示增强后，进入了 Phase 4 的多模型体验建设阶段。当前提供：

- 单图上传入口
- 五种状态反馈：`idle / uploading / running / success / error`
- mockable `predict` client，可在未联通后端时独立演示
- 结果总览、病害列表和量化字段占位
- `/results/demo` 结果页样例
- 模型列表读取与模型选择
- 同图双模型对比
- 历史记录再次模型对比
- 图像级并排对比
- 病害类别级差异对比

当前已验证：

- `vitest` 测试通过
- `eslint` 校验通过
- `next build --webpack` 构建通过
- 页面消费的是统一协议字段，而不是模型原始输出
- 浏览器侧连接真实后端后的上传到结果展示链路已跑通
- 真实 runner 返回结果的前端消费已完成端到端验证
- 前端已可消费 `GET /models` 返回的模型列表
- 前端已可在 `/predict` 请求中传递 `model_version`

当前前端已完成的 Phase 3 / Phase 4 关键能力包括：

- 更完整的结果页展示与联动交互
- overlay / JSON 导出入口
- 历史结果基础回看能力
- 更清晰的错误态与空态展示
- 模型版本选择与对比分析
- 历史结果重新分析与对比
- 结果摘要级、图像级和类别级差异表达
- ✅ 历史记录页面全面重构与UI优化
- ✅ 统计概览栏（总记录、今日新增、检出病害、平均耗时）
- ✅ 高级筛选（关键词搜索、类别筛选、多种排序）
- ✅ 批量操作（选择模式、多选删除、批量导出JSON）
- ✅ 分页支持（每页12条记录）

当前更值得继续推进的是：

- 更清晰的模型可用性提示
- 更强的 mask / overlay 差异表达
- 视需要增加实例级近似匹配对比
- 为 `sliced` 推理和 batch 流程预留更清晰的 UI

## 启动方式

```bash
npm install
npm run dev
```

如需连接真实后端，设置：

```bash
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

未设置时，页面会回退到 mock 结果，便于前后端并行联调。

如果后端返回多个模型版本，前端会优先选择当前实际可用的 active model，并在界面中以“模型名 / 版本”的形式展示。

所有开发任务开始前，应先参考 `AGENTS/` 目录确认当前阶段和执行规范，再继续调整前端实现。
