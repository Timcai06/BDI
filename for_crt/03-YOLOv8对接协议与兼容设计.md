# YOLOv8 对接协议与兼容设计

## 文档目的

本文档用于明确桥梁病害识别系统中，前端、后端与算法模块之间围绕 `YOLOv8` 建立的统一对接协议。

它的目标不是描述某一个具体模型，而是约定一套能够：

- 先支持当前 `YOLOv8` 模型
- 再兼容基于 `YOLOv8` 优化后的算法
- 后续兼容不同推理后端与不同部署形式

的统一规则。

该文档将作为系统开发中的“接口契约”，保证产品开发与算法优化可以并行进行，互不阻塞。

## 一、设计原则

为了让系统真正具备持续演进能力，所有围绕 `YOLOv8` 的对接都应遵循以下原则：

### 1. 面向能力，不面向具体权重文件

系统对接的对象不应是某一个固定的 `.pt` 文件，而应是“一个能够完成桥梁病害检测与分割的推理能力”。

这意味着：

- 当前可以接 `YOLOv8-seg`
- 后续可以接调优版 `YOLOv8-seg`
- 后续也可以接导出的 `ONNX` 或 `TensorRT engine`

只要它们能够按约定返回结果，系统就不需要重构。

### 2. 模型可替换，接口不频繁变化

模型可以不断优化，但系统的输入输出接口应尽可能稳定。

更进一步说，建议把“模型升级”默认视为“配置变化”而不是“接口变化”。

### 3. 前端不感知模型细节

前端只依赖统一结果结构，不关心底层模型使用了哪种权重、哪种后端、是否采用了切片推理。

### 4. 后处理与量化逻辑独立于模型

裂缝长度、宽度、面积等量化逻辑，不应写死在模型推理代码中，而应作为推理后的独立模块存在。

### 5. 为高分辨率和多策略推理预留空间

接口设计要允许后续切换：

- 整图推理
- 切片推理
- 高精度模式
- 高速度模式

## 二、系统中的 YOLOv8 角色定位

在当前阶段，`YOLOv8` 在系统中的角色应被定义为：

**核心初始推理引擎**

也就是说：

- 它是当前产品原型的主推理模型
- 它不是系统唯一的永久实现方式
- 它应被包裹在统一适配层中
- 它的输出需要被标准化后才能进入产品层

这个定位非常重要，因为它决定了：

- 系统前后端可以先围绕 `YOLOv8` 快速落地
- 同时不把整个产品绑死在某一个模型实现上

## 三、统一调用方式

无论后端最终调用的是：

- 原始 `YOLOv8 .pt`
- 调优后的 `YOLOv8 .pt`
- 导出的 `ONNX`
- 导出的 `TensorRT engine`

都建议在系统内部暴露一个统一调用入口。

建议逻辑上统一为：

```text
run_inference(image, config) -> raw_result
```

这里的关键不在于具体函数名，而在于调用模式要统一。

调用入口应做到：

- 输入一张图像或图像路径
- 接收推理参数
- 返回可被后续结果适配层处理的标准原始结果

推荐在系统内部把调用过程拆成两步：

```text
load_runner(model_spec) -> runner
runner.predict(image, infer_spec) -> raw_result
```

这样做的意义是：

- 模型选择与推理执行分离
- 方便缓存已加载模型
- 更适合后续支持多版本模型切换

## 四、输入协议

算法推理层接收的输入建议分为三类：

### 1. 图像输入

用于描述待推理图像本身。

建议包括：

- 图像文件
- 图像路径
- 图像唯一标识
- 原始尺寸信息

### 2. 推理配置

用于描述当前推理过程的策略。

建议包括：

- 置信度阈值
- IoU 阈值
- 输入尺寸
- 是否启用切片推理
- 切片尺寸与重叠率
- 是否返回可视化图
- 是否执行量化计算

### 3. 运行环境信息

用于记录当前结果由什么模型和什么推理方式得到，方便结果管理和后期答辩展示。

建议包括：

- 模型名称
- 模型版本
- 权重路径或模型标识
- 推理后端
- 设备信息

## 五、推荐的输入结构

为了让后端实现更清晰，建议将一次推理请求抽象为如下结构：

```json
{
  "image": {
    "image_id": "bridge_001.jpg",
    "path": "/data/input/bridge_001.jpg"
  },
  "config": {
    "confidence": 0.25,
    "iou": 0.45,
    "imgsz": 1280,
    "inference_mode": "direct",
    "enable_metrics": true,
    "return_overlay": true
  },
  "model": {
    "model_name": "yolov8-seg",
    "model_version": "v1",
    "backend": "pytorch"
  }
}
```

该结构的意义在于：

- 图像信息、推理配置和模型配置被清晰拆开
- 后续切换模型版本时，只需要改 `model` 配置
- 后续切换推理策略时，只需要改 `config`

如果后面要支持批量任务，建议额外保留一个 `task_id` 字段，用于结果追踪和缓存命中。

## 六、原始结果层与业务结果层分离

为了兼容后续优化版算法，必须明确区分两类结果：

### 1. 原始结果层

这是模型推理后直接返回的结果，可能是：

- `boxes`
- `scores`
- `class_ids`
- `masks`
- 原始张量或模型框架对象

对于当前 `Ultralytics` 路线，这一层通常可以理解为官方 `Results` 对象及其内部的 `Boxes`、`Masks` 等结构。

这一层只在模型适配层和结果适配层之间流动。

### 2. 业务结果层

这是后端返回给前端和产品层的结果。

这一层必须稳定、统一、可理解。

这层不应暴露框架内部对象，只应提供标准化后的字段。

推荐默认约束是：

- 不把张量对象直接回传给前端
- 不把模型内部类名暴露给 API
- 不让前端自行解析类别 ID 和 mask 原始格式

## 七、统一输出协议

这是整套设计中最关键的部分。

无论底层模型如何变化，最终输出给前端和系统其他模块的结果结构都应尽可能保持一致。

建议输出至少包含以下字段：

- 图像基本信息
- 推理耗时
- 模型版本信息
- 推理模式信息
- 病害结果列表

每个病害结果至少包含：

- 唯一标识
- 类别名称
- 置信度
- 边界框
- Mask
- 预留量化字段

推荐结构如下：

```json
{
  "schema_version": "1.0",
  "image_id": "bridge_001.jpg",
  "width": 3840,
  "height": 2160,
  "inference_ms": 168,
  "model_name": "yolov8-seg",
  "model_version": "v1",
  "backend": "pytorch",
  "inference_mode": "direct",
  "detections": [
    {
      "id": "damage_001",
      "category": "crack",
      "category_id": 0,
      "confidence": 0.94,
      "bbox": [120, 300, 420, 350],
      "mask": "polygon_or_rle_or_saved_path",
      "metrics": {
        "length_px": null,
        "width_px": null,
        "area_px2": null
      }
    }
  ],
  "artifacts": {
    "overlay_image": null,
    "json_path": null
  }
}
```

建议把输出字段分成四组理解：

- 元信息：图像尺寸、模型版本、后端、协议版本
- 推理信息：耗时、推理模式
- 检测结果：类别、置信度、bbox、mask、量化字段
- 导出信息：叠加图路径、结果文件路径

## 八、Mask 字段的兼容策略

由于不同模型或不同导出后端对 `mask` 的表达方式可能不同，因此建议不要把 `mask` 字段写死成单一格式。

推荐支持以下几种表达方式中的一种或多种：

- polygon
- RLE
- 二值图路径
- 压缩后的 mask 文件路径

更重要的是，前后端应约定一个主展示方式。

我建议：

- 系统内部允许多种 mask 表达方式
- 对前端展示，优先转换成统一可绘制格式

这样后续无论你同学底层如何调整 `mask` 生成逻辑，前端都不用跟着变化。

更具体地说，建议你们内部约定：

- 存储层可以保留原始 mask 表达
- API 输出层优先给前端返回一种稳定格式
- 导出层可以按比赛需要输出单独 mask 文件或 JSON 中的 mask 描述

## 九、类别映射协议

病害类别在模型内部通常以数字类别 ID 表示，但系统层不应直接使用裸 ID。

建议统一维护一份类别映射配置，至少包含：

- 类别 ID
- 类别英文标识
- 类别中文名称
- 展示颜色
- 是否启用该类别

例如：

```json
[
  {
    "id": 0,
    "key": "crack",
    "label": "裂缝",
    "color": "#ff4d4f"
  }
]
```

这样做的好处是：

- 模型内部类别编号变化时，系统层只需要改映射
- 前端展示颜色和名称可以统一管理

## 十、模型兼容设计

如果后续算法同学继续基于 `YOLOv8` 做优化，系统至少应兼容以下变化：

### 1. 权重版本变化

例如：

- `v1`
- `v2`
- `best_small_crack`
- `best_low_light`

### 2. 推理后端变化

例如：

- `pytorch`
- `onnxruntime`
- `tensorrt`

### 3. 推理策略变化

例如：

- `direct`
- `sliced`
- `high_precision`

### 4. 后处理逻辑变化

例如：

- 不同 NMS 策略
- 不同 mask 平滑方式
- 不同面积筛选规则

只要系统内部通过配置和适配层管理这些变化，前端和业务接口都不需要跟着改动。

## 十一、优化后算法的接入原则

当算法同学完成一轮优化后，你接入项目时不应该把它理解为“换一个模型文件”，而应该理解为“注册一个新的模型版本”。

系统真正接入的对象不是某个具体权重文件，而是一个符合统一调用方式和统一结果协议的新推理版本。

这意味着：

- 前端不需要因为新模型而重写
- 后端 API 不需要因为新模型而改路径
- 你真正需要调整的是模型注册信息和结果适配层

## 十二、算法同学每次交付新模型时应提供的内容

为了让接入流程稳定，算法同学每次提供优化后的模型时，不应只给一个权重文件，还应同时给出一份模型说明。

建议每次交付至少包含：

- `model_name`
- `model_version`
- `weights_path`
- `backend`
- `imgsz`
- 类别映射说明
- 是否支持 `mask`
- 是否支持切片推理
- 是否有特殊后处理
- 一组测试图片与参考结果

推荐交付结构如下：

```json
{
  "model_name": "yolov8-seg",
  "model_version": "v2-small-crack",
  "weights_path": "models/v2-small-crack.pt",
  "backend": "pytorch",
  "imgsz": 1280,
  "supports_masks": true,
  "supports_sliced_inference": true
}
```

这样做的意义是：

- 模型优化被视为受控版本升级
- 接入工作从“手动改代码”变成“注册新模型配置”
- 后续结果对比和问题回溯更容易进行

## 十三、你接入优化后算法时真正需要做的事情

在理想结构下，你接入一个优化后的模型版本时，通常只需要做两件事：

### 1. 在模型注册表中增加新版本

把新模型作为一个新的可用配置注册进去，例如：

```python
{
    "model_name": "yolov8-seg",
    "model_version": "v2-small-crack",
    "weights_path": "models/v2-small-crack.pt",
    "backend": "pytorch",
    "imgsz": 1280,
    "supports_masks": True,
    "supports_sliced_inference": True,
}
```

### 2. 检查结果适配层是否仍能输出统一协议

如果优化后的模型仍然返回和当前 `YOLOv8` 兼容的原始结果结构，那么前后端几乎不需要改动。

如果有差异，例如：

- `mask` 表达方式变化
- 类别 ID 变化
- 特殊后处理增加

你只需要修改适配层，不需要重写前端和 API。

## 十四、推荐的接入流程

建议你每次接入新模型版本时，都按同一套流程进行：

1. 获取模型配置与说明
2. 将新模型注册进系统
3. 测试模型是否可成功加载
4. 跑通单张图推理
5. 检查标准化结果字段是否完整
6. 验证前端结果页是否正常展示
7. 验证 JSON 和叠加图是否能正常导出
8. 验证批量流程与历史任务是否正常

这套流程的意义在于：

- 每次优化都走同一套接入路径
- 兼容性问题会被尽早发现
- 系统不会因为一次模型升级而整体失稳

## 十五、接入后的重点验证项

你接入优化后的算法后，最需要关注的不是模型内部细节，而是下面这些工程问题：

- 模型能不能成功加载
- 单张图推理能不能跑通
- 返回字段是否符合统一协议
- 前端页面是否无需改动即可展示
- 导出结果是否正常
- 批量任务是否可正常执行
- 历史任务是否能区分模型版本

也就是说，你真正验证的是“系统兼容性”，而不是重新做一次算法研究。

## 十六、算法优化过程中必须遵守的边界

为了保证系统长期可维护，算法同学在优化过程中应尽量遵守以下边界：

- 不随意改最终输出字段名
- 不让前端直接依赖类别 ID 的语义
- 不让前端直接解析原始 `Results`
- 如果 `mask` 表达变化，在适配层统一解决
- 如果类别映射变化，在映射配置层解决
- 如果增加切片推理、导出后端或特殊后处理，最终输出结构仍应保持一致

一句话说，算法可以持续变强，但产品协议不能被频繁打碎。

## 十七、推荐的模型配置字段

为了避免后面配置越来越乱，建议模型配置至少包含下面这些字段：

- `model_name`
- `model_version`
- `backend`
- `weights_path`
- `device`
- `imgsz`
- `task`
- `supports_masks`
- `supports_sliced_inference`

这组字段足够支撑你们当前比赛阶段的大多数切换需求。

## 十八、模型注册思路

为了便于后续维护，我建议你们不要在代码里到处硬编码模型路径，而是维护一个简单的模型注册表。

模型注册表至少可以记录：

- 模型名称
- 模型版本
- 权重位置
- 推理后端
- 支持的类别集
- 推荐输入尺寸
- 是否支持切片推理

这样做的好处是：

- 方便切换实验模型
- 方便对比不同版本
- 方便比赛答辩时展示“系统支持多版本算法”

如果后面模型多起来，建议把注册表放成单独配置文件，而不是散落在代码中。

## 十九、与前端的对接规则

前端不应直接关心：

- 当前底层是 `.pt` 还是 `TensorRT`
- 当前是否启用了切片推理
- 当前模型是否做了特殊增强

前端只关心：

- 有哪些检测结果
- 每个病害的位置和类别是什么
- 置信度是多少
- Mask 怎么画
- 量化结果是什么
- 推理耗时是多少

因此，前端与后端之间的协议必须始终稳定，不能因为算法升级频繁改字段。

对于前端来说，最重要的永远是这几个字段稳定：

- `detections`
- `category`
- `confidence`
- `bbox`
- `mask`
- `metrics`
- `inference_ms`

## 二十、当前阶段最值得先落地的兼容能力

为了避免一开始设计过重，我建议第一阶段只实现下面这些兼容点：

- 支持 `YOLOv8 .pt`
- 预留 `ONNX` 和 `TensorRT` 的模型配置字段
- 支持 `direct` 和 `sliced` 两种推理模式
- 统一输出 `category`、`confidence`、`bbox`、`mask`、`metrics`
- 允许记录模型版本和推理后端

这个阶段不要求把所有后端都真正跑通，但必须在结构上预留位置。

另外，建议第一阶段就把这两个约束固定下来：

- 结果 JSON 自带 `schema_version`
- 所有错误都返回统一错误结构

推荐错误结构如下：

```json
{
  "error": {
    "code": "MODEL_NOT_AVAILABLE",
    "message": "Requested model version is not registered.",
    "details": null
  }
}
```

这样前端可以稳定处理失败状态，也方便排查联调问题。

## 二十一、为什么这套协议适合你们当前项目

对你们这类“算法同学持续优化 YOLOv8，我来做产品”的组合，这套协议的价值非常直接：

- 算法可以持续迭代，不阻塞前后端
- 前后端可以先做稳定展示，不等待最终模型定稿
- 不同模型版本可以并存，方便对比和回退
- 未来切换到 ONNX 或 TensorRT 时，不需要推倒重来

## 二十二、结论

如果你们要做一个“以 YOLOv8 为核心，同时兼容基于 YOLOv8 优化后算法”的产品，那么真正需要先做好的不是某个页面，也不是某个模型脚本，而是这套统一对接协议。

只要这套协议成立：

- 算法同学可以持续优化模型
- 我可以持续推进前后端产品
- 系统可以不断增强，而不需要频繁推倒重来

因此，这份协议文档应被视为后续系统开发的底层约束之一。
