# 桥梁病害识别系统架构草案

## 文档目的

本文档用于描述当前赛题方向下的系统整体结构、模块关系与接口协作原则。

该架构草案以“先形成完整产品闭环，再逐步接入更强算法能力”为核心思路，适合作为我与算法同学并行开发时的统一参考。

## 架构目标

本系统的目标不是单纯展示 YOLOv8 模型结果，而是构建一个围绕桥梁病害识别任务的完整产品原型。系统应具备以下特点：

- 能接收无人机原始图像作为输入
- 能调用当前 YOLOv8 及后续优化模型完成推理
- 能输出病害类别、Mask、置信度等核心结果
- 能在前端直观展示识别效果
- 能导出结构化结果文件，支撑比赛交付与演示
- 能支持后续扩展病害量化、批量任务和性能优化

## 基于调研的结构判断

结合赛题要求、Ultralytics 官方文档、相关论文和公开项目，我对系统结构的判断进一步收敛为以下几点：

### 1. 产品主线应当先围绕 YOLOv8-seg 建立

当前最合理的起点仍然是 `YOLOv8-seg` 或同类轻量实例分割框架，而不是一开始就围绕重型 Transformer 分割模型构建整套系统。

原因在于：

- YOLO 路线更适合快速形成端到端闭环
- 更容易满足比赛中的推理效率目标
- 更适合作为当前产品原型的稳定核心
- 便于后续由算法同学逐步替换权重、改进结构、优化后处理

### 2. 系统绝不能直接绑定某一个 `.pt` 权重文件

从产品结构角度看，最大的风险不是模型效果不够，而是前后端直接依赖某个具体模型实现，导致后续算法一改，系统就跟着大改。

因此系统必须围绕“统一推理能力”而不是“某个具体 YOLOv8 权重文件”来设计。

### 3. 兼容能力的核心不是“支持更多前端页面”，而是“支持不同推理后端与不同优化版本”

Ultralytics 官方文档说明，YOLO 系列支持预测、分割和导出到多种格式，包括 `ONNX`、`TensorRT` 等推理后端。很多公开 API 项目也采用类似思路，把模型推理封装成独立服务，再通过统一 API 对外暴露结果。

这意味着你们现在最值得提前设计的兼容能力包括：

- 同一任务下不同权重版本切换
- `PyTorch .pt` 与导出后的 `ONNX` / `TensorRT` 模型切换
- 常规整图推理与切片推理切换
- 原始 YOLOv8 输出与后续优化算法输出的统一适配

更具体地说，Ultralytics 官方文档已经明确：

- 预测结果会以 `Results` 对象返回
- 结果中天然包含 `boxes` 与 `masks`
- 官方支持 `plot()`、`save()`、`save_txt()` 等结果处理方式
- 官方支持将模型导出为 `ONNX`、`TensorRT engine` 等格式

这说明你们当前的系统完全可以建立在“官方结果对象 -> 内部标准结构”这一条明确链路之上，而不是自己凭空设计一套脱离生态的推理流程。

### 4. 4K 图像和小病害场景，决定了系统需要预留“切片推理策略”

公开研究和项目都反复强调，高分辨率大图中的小目标检测是核心难点。`SAHI` 这类切片推理方法的目标就是提升大图中的小目标检测效果，尤其适用于无人机和航拍场景。

这意味着系统在最初结构上就应当允许：

- 普通整图推理
- 滑窗/切片推理
- 后续的多尺度推理或 patch-based 推理

而不是把推理流程写死成“读图 -> 单次模型前向 -> 返回结果”。

## 总体架构思路

系统整体建议采用“前端展示层 + 后端服务层 + 算法推理层”的三层结构。

### 1. 前端展示层

前端负责承接用户交互与结果展示，是系统面向评审和演示的主要窗口。

建议承担以下职责：

- 图像上传与任务提交
- 任务状态展示
- 原图、Mask 与结果叠加可视化
- 病害类别、置信度、几何量化结果展示
- 结果筛选、阈值调节与详情查看
- 结果导出与演示流程呈现

### 2. 后端服务层

后端是系统的中枢层，负责连接前端交互与算法推理。

建议承担以下职责：

- 接收图像与请求参数
- 调用算法推理模块
- 统一封装算法输出格式
- 管理结果文件与日志
- 提供单图推理、批量推理、结果查询与下载接口
- 隔离前端与模型实现细节

### 3. 算法推理层

算法推理层由算法同学重点负责，以 YOLOv8 及其后续优化版本为核心。

建议承担以下职责：

- 模型加载
- 图像预处理
- 目标检测与实例分割
- 置信度计算
- 基础后处理
- 输出标准化前的原始结果生成

## 推荐的模块划分

为了保证后续维护与升级更顺畅，建议将系统拆成以下几个逻辑模块：

### 一、输入管理模块

负责管理图像输入及参数配置。

主要内容包括：

- 单张图像上传
- 批量图像上传
- 推理参数传递
- 任务编号生成

### 二、推理调度模块

负责将前端请求转换为可执行的算法调用流程。

主要内容包括：

- 图像读取与校验
- 推理任务触发
- 同步或异步处理控制
- 推理耗时记录
- 推理策略选择

这里建议把“推理策略”单独抽象出来，而不是写死在接口实现中。

建议至少预留两种策略：

- `direct_inference`：直接整图推理，适合 MVP 和常规图像
- `sliced_inference`：切片推理，适合 4K 图像和微小病害场景

后续如果算法同学加入更复杂的高分辨率策略，也只需要扩展推理策略层，而不需要重写整个系统。

### 二点五、模型适配模块

这是为了满足“以 YOLOv8 为核心，同时兼容后续优化算法”的关键模块，建议从一开始就单独存在。

该模块的职责不是做业务后处理，而是统一管理“模型如何被调用”。

建议它承担以下职责：

- 管理当前激活的模型版本
- 区分不同模型格式，如 `.pt`、`.onnx`、`.engine`
- 区分不同推理后端，如 `PyTorch`、`ONNX Runtime`、`TensorRT`
- 暴露统一的推理入口
- 返回统一的原始推理结果对象

从系统设计上看，它相当于一层 `model gateway` 或 `runner adapter`。

这样后续即使你同学把当前 YOLOv8 模型优化成：

- 更换骨干网络
- 增加定制后处理
- 导出为 TensorRT
- 改成 YOLOv8 的优化变体

前后端也仍然只和这层模块交互。

建议这层内部最少拆成两个角色：

- `ModelRegistry`
  - 记录可用模型、版本、后端、输入尺寸、能力标签
- `ModelRunner`
  - 真正执行推理，屏蔽 `.pt`、`.onnx`、`.engine` 之间的差异

这样做的好处是：

- 方便快速切换不同实验模型
- 方便记录比赛过程中不同版本的表现
- 方便后面接入导出后端时不改业务逻辑

### 三、结果适配模块

这是我负责系统设计时最关键的一层。

它的作用是把 YOLOv8 或后续优化模型的原始输出，统一转换为系统内部的标准结果格式。这样可以保证前端和后端不直接依赖具体模型实现。

该模块建议负责：

- 解析类别信息
- 解析置信度
- 解析边界框
- 解析 Mask
- 生成统一 JSON 结构
- 预留病害量化字段

对于你们当前项目，这一层必须做到两件事：

- 兼容当前 YOLOv8 原始输出
- 兼容未来基于 YOLOv8 优化后的输出差异

也就是说，不管算法同学之后是否改变：

- 类别映射方式
- mask 表示方式
- bbox 生成逻辑
- 推理后处理逻辑

最终进入前端和业务层的结果都要保持稳定。

### 四、可视化渲染模块

负责把结果转换为适合前端展示和演示的内容。

主要内容包括：

- 原图与 Mask 叠加
- 病害颜色区分
- 检测框与标签绘制
- 局部结果高亮
- 导出可视化图片

### 五、结果管理模块

负责保存、查询和导出识别结果。

主要内容包括：

- 结果 JSON 存储
- 可视化图片存储
- 日志记录
- 批量任务结果整理
- 对外下载接口

### 六、病害量化模块

这是后续可持续增强的重点模块，建议在前期就预留位置。

主要内容包括：

- 裂缝长度估计
- 裂缝宽度估计
- 面积类病害面积估计
- 基于像素比例的实际尺度换算

该模块前期可以先保留字段，后期逐步增强。

## 系统数据流

建议系统整体按如下流程运转：

1. 用户在前端上传无人机图像并设置参数
2. 前端请求后端推理接口
3. 后端调用算法推理层执行 YOLOv8 推理
4. 原始推理结果进入结果适配模块
5. 后端生成统一结构化结果
6. 可视化模块生成叠加图或展示数据
7. 前端展示识别结果、耗时和导出内容

这个流程的核心思想是：前端看的是统一结果，算法层负责不断变强，但不直接影响产品层结构。

如果按更细的工程流来看，建议明确为：

1. 图像进入输入管理模块
2. 推理编排层选择模型版本与推理策略
3. 模型适配层调用具体 YOLOv8 运行器
4. 原始 `Results` 对象进入结果适配层
5. 结果适配层提取 `boxes`、`masks`、`conf`、`cls`
6. 量化模块补充长度、宽度、面积等字段
7. 可视化模块生成叠加图和导出结果
8. 前端消费统一 JSON 与叠加资源

## 推荐的产品结构定稿

基于这轮调研，我认为你们的产品结构应当进一步明确为“五层式”而不是简单三层式。

### 第一层：前端交互层

负责：

- 图像上传
- 参数设置
- 结果展示
- 任务状态查看
- 结果下载与演示

### 第二层：业务 API 层

负责：

- 接收请求
- 参数校验
- 路由组织
- 任务编排
- 权限和日志的基础管理

### 第三层：推理编排层

负责：

- 决定使用哪个模型版本
- 决定使用整图还是切片推理
- 决定是否启用导出后端
- 记录推理耗时和运行环境

这一层是“调度脑”，不直接关心前端，也不直接关心模型内部实现。

### 第四层：模型适配层

负责：

- 把 `YOLOv8 .pt`
- 把导出的 `ONNX`
- 把导出的 `TensorRT engine`
- 把后续基于 YOLOv8 的定制优化模型

统一包装成相同调用方式。

### 第五层：结果与可视化层

负责：

- 统一结果协议
- 结果 JSON 生成
- 可视化叠加图生成
- 报告和导出
- 病害量化计算

这五层结构最重要的价值在于：

- 前端稳定
- 后端可控
- 模型可替换
- 推理后端可切换
- 结果表达可持续增强

## 推荐的最小工程目录思路

为了让这套结构更容易落地，我建议你们在工程上也尽量按职责分目录，而不是把 API、模型调用和结果处理混在一起。

推荐按下列思路组织：

- `frontend/`
  - 页面、组件、结果展示逻辑
- `backend/api/`
  - 路由、请求校验、任务接口
- `backend/services/`
  - 推理编排、结果管理、报告导出
- `backend/models/`
  - `registry`、`runner`、`adapters`
- `backend/postprocess/`
  - 结果标准化、量化、可视化
- `artifacts/`
  - 原图、JSON、可视化图、日志

这不是唯一目录结构，但核心原则必须保持：

- API 不直接操作模型细节
- 模型运行不直接耦合展示逻辑
- 量化和结果导出不直接写进推理脚本

## 推荐的运行形态

从你们当前比赛阶段看，建议先按“两种运行形态”设计系统，而不是一开始就上复杂微服务。

### 形态一：单体原型

适合比赛前期。

特点：

- 前端单独运行
- 后端同时承担 API、推理编排、模型适配和结果处理
- 最快形成 MVP

### 形态二：逻辑分层但物理上仍可同机部署

适合比赛中后期。

特点：

- 保持代码职责清晰
- 模型适配层可以单独维护
- 后续如果需要切到更高性能机器或独立 GPU 服务，更容易拆分

对你们当前双人团队来说，我不建议一开始就做重型分布式架构，但我强烈建议从代码结构上为拆分预留空间。

## 兼容 YOLOv8 优化算法的关键设计要求

如果系统要真正兼容“基于 YOLOv8 优化后的算法”，我建议从一开始就满足以下要求。

### 1. 模型调用接口统一

无论底层是：

- 原始 YOLOv8-seg
- 调参后的 YOLOv8-seg
- 加入额外后处理的 YOLOv8 变体
- 导出后的 ONNX 或 TensorRT 模型

系统都应通过同一个调用接口发起推理。

### 2. 结果结构统一

前端只接收统一结构，不关心模型内部细节。

### 3. 模型版本可配置

系统里应允许记录：

- 模型名称
- 模型版本
- 权重路径
- 推理后端
- 输入尺寸
- 是否启用切片推理

这样既方便联调，也方便答辩时展示“不同版本效果对比”。

### 4. 推理策略可配置

用户或后端策略应允许切换：

- 常规模式
- 高精度模式
- 高分辨率模式

从产品语言上，这比直接暴露“SAHI 开关”“TensorRT 开关”更适合后续展示。

### 5. 后处理与量化模块独立

长度、宽度、面积等量化逻辑不要写进模型调用代码里，应放在模型之后的独立模块中。这样模型升级时，量化逻辑不需要重写。

### 6. 结果协议要带版本号

如果后面你们补充更多字段，例如严重程度、病害分组、构件关联信息，建议结果 JSON 自带协议版本。

例如：

- `schema_version: "1.0"`

这样未来即使扩展字段，也能避免前后端联调混乱。

### 7. 原始结果缓存与重放能力

为了方便调试和答辩演示，建议系统允许保存一次推理后的结构化结果和可视化输出。

这样做有几个好处：

- 不必每次演示都重新跑模型
- 可以稳定复现实验样例
- 可以对同一张图做不同模型版本对比

这对于比赛场景非常实用。

## 我建议的第一版工程落点

在现阶段，为了既保留兼容性又不把系统做得过重，我建议你们第一版先按下面的结构落地：

- 前端：上传图像、结果页、病害详情面板、导出按钮
- 后端 API：`/predict`、`/predict-batch`、`/results/{id}`、`/health`
- 推理编排：支持 `direct` 和 `sliced` 两种模式
- 模型适配：先支持 `YOLOv8 .pt`，接口上预留 `onnx` 和 `engine`
- 结果适配：统一输出 `category`、`confidence`、`bbox`、`mask`、`metrics`
- 结果管理：保存原图、可视化图、JSON 结果

这个版本的目标不是一步到位，而是先把“模型可替换的产品骨架”搭出来。

如果只保留一句最重要的工程要求，那就是：

**前端调用的是产品接口，后端调用的是统一推理接口，而不是直接调用某个 YOLOv8 权重文件。**

## 为什么我认为这是当前最稳的结构

从调研结果看，这个结构有几个明显优势：

- 与当前 YOLOv8 路线完全兼容
- 允许后续平滑切换到更快的 ONNX 或 TensorRT 推理
- 适合引入切片推理来处理 4K 图像和小病害
- 前端和业务层不会被算法频繁改动拖垮
- 很适合比赛中边开发边演示边迭代

## 建议的接口协作原则

为了便于与算法同学并行协作，系统必须围绕统一接口推进。

### 输入侧约定

建议所有算法推理调用都接受以下基础信息：

- 图像文件或图像路径
- 置信度阈值
- 需要返回的结果类型
- 可选的任务信息

### 输出侧约定

建议所有模型结果最终统一整理为系统内部标准结构，至少包含：

- 图像基本信息
- 推理耗时
- 模型版本信息
- 病害结果列表
- 每个病害的类别
- 每个病害的置信度
- 每个病害的 bbox
- 每个病害的 mask
- 预留量化字段

示意结构如下：

```json
{
  "image_id": "bridge_001.jpg",
  "width": 3840,
  "height": 2160,
  "inference_ms": 168,
  "model_version": "yolov8-seg-v1",
  "detections": [
    {
      "id": "damage_001",
      "category": "crack",
      "confidence": 0.94,
      "bbox": [120, 300, 420, 350],
      "mask": "polygon_or_rle_or_saved_path",
      "metrics": {
        "length_px": null,
        "width_px": null,
        "area_px2": null
      }
    }
  ]
}
```

该结构的意义在于：

- 前端只依赖这套标准结构
- 后端只负责转换与分发
- 算法可以持续优化，而不频繁冲击产品层

## 我负责的核心架构职责

在该架构下，我主要承担以下职责：

- 设计前端与后端的产品骨架
- 设计统一 API 与结果格式
- 设计算法接入适配层
- 完成前端展示与后端服务联动
- 为后续算法增强和量化模块留出扩展空间

我的目标不是决定模型内部如何优化，而是保证模型能力可以被稳定、清晰、完整地接入系统。

## 与算法同学的接口关系

在协作上，算法同学和我的工作关系应当是：

- 算法同学输出“可调用的推理能力”
- 我输出“可使用的系统能力”

也就是说：

- 模型训练和精度提升主要由算法同学负责
- 模型如何接入产品、如何展示、如何导出、如何形成演示闭环由我负责

只要双方遵循统一接口，算法优化和产品开发就可以并行推进。

## 当前阶段的架构原则

当前阶段不追求一次性搭建完整复杂系统，而是坚持以下原则：

- 先完成最小闭环
- 先保证可演示、可联调
- 先保证模型可替换
- 先保证前后端结构清晰

在这个基础上，后续再逐步增强：

- 批量处理能力
- 性能统计能力
- 报告导出能力
- 病害量化能力
- 更复杂的任务管理能力

## 总结

该系统架构草案的核心思想是：以 YOLOv8 为当前算法起点，以统一接口为协作基础，以前后端产品骨架为主要承载方式，逐步构建一套既能承接算法迭代、又能服务比赛展示与实际应用表达的桥梁病害识别系统原型。

更具体地说，现阶段最适合你们的不是围绕“某个 YOLOv8 权重文件”搭产品，而是围绕“一个可兼容 YOLOv8 及其优化变体的推理产品框架”搭产品。
