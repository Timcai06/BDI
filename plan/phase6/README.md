# Phase 6 执行说明

## 目标

本阶段目标是把系统从“功能可运行”优化为“可稳定承接真实巡检业务”的工程化版本。

核心不再是新增页面数量，而是提升业务稳定性、决策可用性和交付可复制性。

## 开工前提

进入本阶段前，默认以下事项已经完成并验收：

- `Phase 5` 已完成核心链路：批次接入、异步任务、结果落库、检索、复核、预警。
- 前端企业化主导航已经建立，`ops` 流程可作为默认业务入口。
- 数据库主存储已切换到 `PostgreSQL`，迁移机制已建立。

## 范围

本阶段优先承诺落地以下能力：

- 任务与状态链路稳定性优化（幂等、重试、超时、失败分类）
- 标准化评级与处置规则引擎（先对齐中国 JTG 口径）
- 关键告警机制（critical finding）与处置时效追踪
- 运营效率优化（高风险优先视图、批量操作、积压可视化）
- 对外集成最小能力（工单接口与结构化导出）

## 实施顺序

1. 先稳后端任务链路和数据库一致性，再扩展规则引擎。
2. 再做决策层能力：评级、优先级、critical finding 模板。
3. 再做前端运营效率优化：待办优先级、批量处置、时效面板。
4. 最后补对外集成接口，验证可对接外部 BMS/EAM 系统。

## 关键约束

- 不新增重型功能（如复杂审批流、视频全链解析）作为本阶段主目标。
- 所有优化必须有量化指标，避免“主观感觉变好”。
- 规则引擎必须版本化，避免改规则后不可追溯。
- 高风险事件必须具备明确的状态与时限字段。

## 完成定义

满足以下条件视为 `Phase 6` 完成：

- 批次任务在高并发下可稳定执行，失败可重试且可追踪。
- 系统能输出标准化评级与优先级建议。
- 关键告警具备从触发到处置的时效追踪闭环。
- 前端可明确呈现“高风险待处置、复核积压、告警时效”。
- 对外提供最小可用工单/导出接口，具备集成基础。
