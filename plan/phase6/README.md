# Phase 6 执行说明

## 目标

本阶段目标是把系统从“功能可运行”升级为“企业可稳定使用”。

重点不是继续堆页面，而是让企业用户在日常巡检里做到三件事：

- 稳定处理大量批次与任务
- 快速定位高风险并完成处置
- 能对接企业现有系统并沉淀可审计数据

## 基于当前现状的判断

已具备基础能力（来自当前代码状态）：

- 后端已有 `batch -> task -> result -> detection -> review -> alert` 主链路
- `PostgreSQL + SQLAlchemy + Alembic` 已接入，支持结构化查询
- 前端已有 `ops` 多页面结构（总览、批次、检索、复核、告警、设置）
- 历史能力已存在，但入口层级与 `ops` 流程需要进一步统一

现阶段主要短板：

- 可观测性不够（成功率、时延、积压、恢复率缺统一基线）
- 决策层不够标准化（评级/优先级/SLA 建议未形成规则版本）
- 运营视图偏“展示”，需进一步转“处置驱动”
- 对外系统集成边界未明确成可联调规范

## 范围

本阶段聚焦以下五个企业可用能力：

- 任务链路可靠性：幂等、重试、超时、积压治理
- 标准化决策引擎：评级、优先级、建议时限（先对齐 JTG）
- 告警时效闭环：critical finding、SLA、升级策略
- 运营体验重构：高风险优先路径、批量处置、历史追溯入口
- 对外交付能力：工单接口、导出规范、联调契约

## 实施顺序

1. 建立指标基线并修复任务稳定性瓶颈。
2. 落地规则引擎与版本管理，打通结果到决策。
3. 重构前端处置路径，确保“先看高风险，再做批量动作”。
4. 固化历史追溯与审计链路，补齐企业留痕要求。
5. 完成最小外部集成接口，验证可对接 BMS/EAM/工单系统。

## 关键约束

- 不脱离现有代码基线重做架构，以增量优化为主。
- 每个优化项必须绑定量化验收指标。
- 规则引擎必须可版本追溯，禁止“隐式改规则”。
- 所有关键状态变更必须可审计（谁、何时、改了什么）。

## 完成定义

满足以下条件视为 `Phase 6` 完成：

- 批次链路在并发场景可稳定运行，失败任务可恢复且可定位原因。
- 系统可稳定输出可解释的风险等级与处置优先级。
- critical finding 可触发、分派、跟踪、升级、关闭并记录 SLA。
- 前端形成企业可用操作流：总览发现风险 -> 批量处置 -> 历史复盘。
- 历史记录与当前 `ops` 入口统一，不再出现“能力在但入口难找”。
- 工单/导出接口可用于真实联调，字段字典与错误码完整。
