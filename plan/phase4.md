# Phase 4

## 阶段主题

兼容优化模型与批量能力建设

## 阶段目标

让系统具备承接算法同学后续优化模型的能力，同时补齐批量处理和高分辨率推理能力。

本阶段的核心不是“先多做几个功能”，而是先把系统从“当前可跑通 `YOLOv8`”升级为“算法可插拔、协议稳定的推理框架”。

## 主要任务

1. 引入模型版本注册机制
   - 明确 `model_name`、`model_version`、`backend`、`weights_path` 等配置字段
   - 支持把同学后续交付的模型视为“新版本注册”而不是“临时替换文件”

2. 补齐统一 runner / adapter 骨架
   - 让后端围绕统一推理入口调用模型
   - 把不同权重、不同后端和不同推理框架的差异收敛在适配层

3. 稳定统一结果协议
   - 确保前端、导出、历史记录和复核流程只依赖标准结果结构
   - 确保模型升级被视为配置变化或适配层变化，而不是接口变化

4. 支持切换不同 `YOLOv8` 版本
   - 优先支持同类优化模型的低成本切换
   - 优先实现“换权重即可切换同类优化模型”的能力

5. 支持 `direct` 和 `sliced` 推理模式
   - 为高分辨率和小病害场景预留切片推理能力

6. 增加批量任务创建与查询
   - 在统一推理骨架稳定后，再补齐批量流程

7. 验证优化后模型的接入流程
   - 形成标准化接入步骤和验收流程

8. 完善历史任务与版本区分
   - 让结果能够区分模型版本、推理模式和产物来源

## 推荐执行顺序

1. 先完成模型注册机制
2. 再完成统一 runner / adapter
3. 再固化统一结果协议
4. 然后接入至少两个模型版本
5. 最后再推进批量任务和 `sliced` 推理

这样做的原因是：

- 如果模型接入层没有先抽象出来，批量和切片能力后面很容易写成一次性逻辑
- 如果结果协议没有先稳定，多模型切换会直接冲击前端和导出能力
- 先完成推理骨架，再补充功能，会更符合“算法可插拔”的目标

## 阶段产出

- 可切换模型版本的系统
- 模型注册与 runner 适配骨架
- 稳定的统一结果协议
- 批量任务基础流程
- 高分辨率推理模式预留
- 优化模型接入流程文档化

## 验收标准

- 系统可以接入至少两个模型版本
- 切换模型时前端无需重构
- 同类优化模型可以通过替换权重和少量配置完成切换
- 差异更大的新算法可以通过适配层接入，而不破坏主流程
- 批量任务可创建、查询、导出
- 优化后的模型可按统一流程接入
