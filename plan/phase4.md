# Phase 4

## 阶段主题

兼容优化模型与批量能力建设

## 阶段目标

让系统具备承接算法同学后续优化模型的能力，同时补齐批量处理和高分辨率推理能力。

本阶段的核心不是“先多做几个功能”，而是先把系统从“当前可跑通 `YOLOv8`”升级为“算法可插拔、协议稳定的推理框架”。

## 当前状态

`Phase 4` 已启动，并已完成第一轮多模型骨架建设。

已完成：

- 模型版本注册机制基础骨架
- `ModelSpec` / `ModelRegistry` / `RunnerManager`
- `/predict` 按 `model_version` 选择 runner
- `GET /models` 模型列表接口
- 前端模型版本选择
- 同图双模型对比
- 历史记录再次模型对比
- 图像级并排对比
- 病害类别级差异对比
- 模型系统关系说明文档
- 新模型接入操作手册

当前仍在推进：

- 接入第二个真实模型版本
- 验证同类模型换权重接入流程

已完成验收：

- `inference_mode` 参数 (direct/sliced) 已支持
- `supports_sliced_inference` 字段已实现
- 模型可用性 (`is_available`) 检查已实现
- `sliced` 推理模式预留接口已完成

## 主要任务

1. 引入模型版本注册机制
   - 当前状态：已完成基础骨架
   - 明确 `model_name`、`model_version`、`backend`、`weights_path` 等配置字段
   - 支持把同学后续交付的模型视为“新版本注册”而不是“临时替换文件”

2. 补齐统一 runner / adapter 骨架
   - 当前状态：已完成第一轮
   - 让后端围绕统一推理入口调用模型
   - 把不同权重、不同后端和不同推理框架的差异收敛在适配层

3. 稳定统一结果协议
   - 当前状态：已完成并持续维护
   - 确保前端、导出、历史记录和复核流程只依赖标准结果结构
   - 确保模型升级被视为配置变化或适配层变化，而不是接口变化

4. 支持切换不同 `YOLOv8` 版本
   - 当前状态：已完成基础切换能力，待真实第二模型验证
   - 优先支持同类优化模型的低成本切换
   - 优先实现“换权重即可切换同类优化模型”的能力

5. 支持 `direct` 和 `sliced` 推理模式
   - 当前状态：待推进
   - 为高分辨率和小病害场景预留切片推理能力

6. 增加批量任务创建与查询
   - 当前状态：待推进
   - 在统一推理骨架稳定后，再补齐批量流程

7. 验证优化后模型的接入流程
   - 当前状态：已补文档，待真实第二模型验收
   - 形成标准化接入步骤和验收流程

8. 完善历史任务与版本区分
   - 当前状态：已完成基础版本区分，待继续增强
   - 让结果能够区分模型版本、推理模式和产物来源

## 推荐执行顺序

1. 先接入第二个真实模型版本
2. 再验证“换权重 + 少量配置”的真实流程
3. 再推进更清晰的模型可用性提示
4. 然后推进批量任务和 `sliced` 推理
5. 最后再增强更细粒度的差异表达

这样做的原因是：

- 目前模型接入层和统一协议已经有了第一轮骨架，下一步该用真实第二模型验证这套抽象是否足够稳
- 如果不先做真实模型验证，后面的批量和切片能力仍可能建立在未充分验证的假设之上
- 在真实切换路径成立之后，再补更多功能，会更符合“算法可插拔”的目标

## 阶段产出

- 可切换模型版本的系统
- 模型注册与 runner 适配骨架
- 稳定的统一结果协议
- 批量任务基础流程
- 高分辨率推理模式预留
- 优化模型接入流程文档化
- 至少两个真实模型版本的接入验证

## 验收标准

- 系统可以接入至少两个模型版本
- 切换模型时前端无需重构
- 同类优化模型可以通过替换权重和少量配置完成切换
- 差异更大的新算法可以通过适配层接入，而不破坏主流程
- 批量任务可创建、查询、导出
- 优化后的模型可按统一流程接入
