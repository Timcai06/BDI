# Phase 5

## 阶段主题

真实巡检业务流程系统化

## 阶段目标

将系统从“单图识别与展示原型”升级为“能够支撑真实企业巡检流程的桥梁病害智能巡检系统原型”。

本阶段不再把重点放在单张图识别结果的展示强化上，而是把识别能力组织成可持续接入、可异步执行、可追踪、可检索、可复核、可预警的系统能力层。

## 当前状态

`Phase 5` 以业务流程重构为主线重新定义。

当前已经具备：

- 双模型融合识别路径
- 单图识别闭环
- 结果展示、导出和历史查看能力

当前尚未具备：

- 以 `batch` 为主语的领域模型
- 异步任务执行与追踪
- 检测记录入库与检索
- 人工复核与预警事件链路

## 主要任务

1. 重构后端领域模型
   - 引入 `bridge / inspection_batch / media_asset / batch_item / inference_task / inference_result / detection / review_record / alert_event` 作为核心对象。
   - 把结构化元数据与检测记录从文件系统迁移到数据库管理。
   - 保持原图、叠加图、结果 JSON 等产物继续可落本地文件系统。

2. 建立批量接入与异步执行链路
   - 支持创建批次并批量上传图片。
   - 上传即入队，由本地 worker 异步执行识别。
   - 支持任务状态查询、失败重试和批次聚合统计。

3. 固化识别结果的标准化落库
   - 识别完成后生成 `result` 头信息。
   - 将病害记录展开为可检索的 `detections` 明细。
   - 为后续规则预警、复核和趋势分析提供统一数据基础。

4. 建立企业化 dashboard 骨架
   - 增加批次列表页、批次详情页、图片详情页和病害检索页。
   - 前端围绕批次进度、筛选、详情和复核组织页面，不再围绕单图 demo 组织。
   - 使用后端聚合统计接口直接驱动 dashboard。

5. 预留并落地初版预警与复核
   - 支持规则型预警事件生成、确认和关闭。
   - 支持人工复核确认、驳回和修正。
   - 形成从识别到预警到人工复核的业务闭环。

## 推荐执行顺序

1. 先定数据库和表结构，再定 API。
2. 先实现批次接入、任务执行和结果落库，再扩前端。
3. 先做规则型预警和基础复核，再考虑更复杂的趋势告警。
4. 继续把双模型融合当作底层能力维护，而不是当前阶段主轴。

## 阶段产出

- 一套以 `batch workflow` 为核心的后端领域模型和数据库结构。
- 面向批次接入、任务追踪、结果查询、病害检索、复核和预警的 API。
- 本地 worker 异步执行能力。
- 面向企业巡检流程的 Dashboard 页面骨架。
- 初版规则预警和人工复核闭环。

## 验收标准

- 可以创建桥梁巡检批次并接收多张图片。
- 批次中的图片能够自动入队并由异步任务完成识别。
- 识别结果和检测记录能够结构化入库并支持查询。
- 前端能够查看批次状态、单张详情、病害筛选和复核状态。
- 系统能够基于规则生成并追踪预警事件。
