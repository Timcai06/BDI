# BDI Dashboard UI 与输出内容优化建议

## 目标

让 `/dashboard` 从“可用的后台页面”升级为“适合比赛展示的 AI 工作台”，同时满足两件事：

1. 与 landing page 的视觉语言保持一致。
2. 更贴合赛题对“多类别识别、像素级分割、量化输出、抗干扰、可用于报告生成”的要求。

赛题来源：
https://www.fwwb.org.cn/topic/show/34d4c23b-0102-4697-82dc-ad0d0e5de04c

## 当前问题

### 1. 页面像后台，不像比赛 demo

- 左中右三列视觉权重太平均，主结果不够突出。
- 当前更像“功能集合”，不像“AI 识别舞台”。
- 用户第一眼看不到最重要的信息：有没有病害、病害在哪、接下来该做什么。

### 2. 与 landing page 风格断层

- landing page 是蓝青高光、玻璃感、叙事型标题、未来感工作台。
- dashboard 虽然延续了深色底，但卡片组织更像传统控制台。
- 字体节奏、标题语气、按钮样式、面板层次没有完全继承 landing 的语言。

### 3. 输出内容没有打中赛题重点

- 当前结果页主要在展示数量、置信度、耗时、原图。
- 赛题更看重：病害类别、Mask、量化参数、抗干扰能力、结构化输出、报告支撑能力。
- 代码中已有 `mask`、`length_mm`、`width_mm`、`area_mm2` 等字段，但页面表达不够强。

### 4. 空结果状态不够专业

- “未识别到明确病害”现在更像兜底提示。
- 对比赛展示来说，空结果也应体现系统判断能力，而不是“没识别出来”。

## 设计目标

识别页应该围绕这三个核心问题展开：

1. 系统判断了什么。
2. 证据在哪里。
3. 结果如何进入后续复核和报告流程。

页面体验要从“后台操作页”转成“智能识别结果页”。

## 建议的信息架构

建议改成 `主舞台 + 结论区 + 次级工具区`。

### A. 主舞台：图像证据区

放在页面中心，占据最大视觉面积。

应包含：

- 原图 / 分割叠加图切换
- 默认优先展示叠加图，而不是原图
- 当前选中病害高亮
- 病害标签和定位
- 模型版本、时间戳、结果状态作为顶部辅助信息

目标：

- 第一眼就看到“系统识别了哪里”
- 强调像素级分割，而不是只有框选

### B. 结论区：结果摘要区

放在主舞台右侧，作为最强信息卡。

应包含：

- 识别结论
- 主要病害类型
- 风险等级
- 下一步建议
- 关键指标矩阵

推荐文案结构：

- 标题：`识别结论`
- 主句：`当前阈值下未识别到明确病害`
- 辅句：`图像清晰度、拍摄角度或病害尺度可能影响微小裂缝召回，建议降低阈值或补充近距离样本后复检。`

如果有病害，则用：

- 主句：`本次识别发现 3 处病害，主要为裂缝与剥落`
- 辅句：`主梁右侧区域风险较高，建议优先复核并导出结构化结果。`

### C. 次级工具区：筛选、系统状态、模型对比

这些内容不应与主结论同级抢视觉。

建议处理方式：

- 系统状态缩成顶部 chip 或小卡
- 筛选器并入右侧工具面板
- 模型对比折叠到下方，作为“深入分析”
- 历史记录改为 drawer、tab 或二级页面入口

## UI 美化方向

## 1. 统一为 landing page 的视觉体系

继续沿用 landing page 已经建立的风格：

- 深黑蓝背景
- 蓝青色辉光
- 大圆角玻璃卡片
- 小号 section label
- 渐变标题
- 克制的动效和高光

不要在 dashboard 再建立一套“纯后台风”的新语言。

## 2. 建立“一个主舞台”

当前页面太碎，应该突出中间主图。

建议：

- 中央图像区放大 20% 到 30%
- 让识别图成为真正视觉中心
- 右侧卡片围绕主图服务，而不是平铺多个同权面板

可以理解为：

- 图像证据是主角
- 识别结论是海报文案
- 筛选和参数只是辅助工具

## 3. 强化层次，不要平均发力

建议按三级层次设计：

- 一级：主图卡、结果结论卡
- 二级：关键指标、病害详情、模型对比
- 三级：系统状态、筛选器、参数信息

避免所有卡片都一样大、一样亮、一样重要。

## 4. 背景要有氛围，不要只是黑

建议保持 landing page 那种低对比的气氛背景：

- 顶部或左上角蓝色 radial glow
- 右下角青色雾化 glow
- 很轻的 grid texture 或 noise
- 透明度保持低，不抢内容

目标是让页面更像“产品界面”，而不是“纯黑后台”。

## 5. 按钮与控件品牌化

当前按钮能用，但产品感还不够。

建议统一成三类：

- 主按钮：蓝青渐变，高亮边框，用于“重新检测”“开始对比”
- 次按钮：玻璃暗底，用于“历史记录”“查看原图”
- 功能按钮：小型 pill，用于“导出 JSON”“导出叠加图”

筛选器建议：

- 更细边框
- 更强 hover/focus 态
- 轻量 glow
- 不要太像默认表单控件

## 6. 空状态必须设计成“专业判断”

空结果状态不是失败，而是系统判断的一种结果。

推荐样式：

- 图标或状态点：绿色或蓝青色
- 标题：`当前未发现明确病害`
- 说明：`系统未在当前阈值下识别到裂缝、剥落、锈蚀或泛碱等目标。建议结合图像质量与拍摄角度进一步复检。`
- 操作建议标签：
  - `降低阈值后复检`
  - `切换模型对比`
  - `查看原图细节`
  - `保留当前记录`

## 输出内容优化

## 1. 结果摘要必须更像工程表达

避免只写“识别完成，返回 N 条结果”。

应改为：

- `本次识别发现 4 处病害，主要为裂缝与剥落，主梁中段风险较高。`
- `系统已输出病害类别、置信度、掩膜与几何参数，可继续用于复核和报告生成。`

## 2. 病害详情必须体现“精细化”

每个病害卡建议至少展示：

- 病害类别
- 置信度
- 所在区域
- 是否有 Mask
- 长度
- 宽度
- 面积
- 复核优先级

这样才能体现赛题里的“精细化智能检测”。

## 3. 默认强调分割结果

赛题强调像素级分割，因此：

- 默认视图建议为叠加图
- 原图作为辅助切换
- 病害高亮尽量体现 mask，而非只靠 bbox

## 4. 增加抗干扰叙述

结果页可以加入一个小模块或一句文案：

- `系统已对阴影、污渍、接缝等非病害区域进行干扰抑制`

如果当前算法层还没有专门输出该信息，也可以先作为展示位，为后续增强预留。

## 推荐页面结构

### 顶部

- 页面标题：`桥梁病害识别结果`
- 副标题：`输出病害类别、分割掩膜、量化参数与复核建议`
- 辅助标签：`Phase 3`、`Model Active`、`Report Ready`

### 中部主区

- 左：识别图主舞台
- 右：识别结论卡 + 指标卡 + 下一步建议

### 下部

- 病害详情列表
- 模型对比
- 结构化输出说明

### 历史与筛选

- 作为次级操作入口，不与主结果同层级展开

## 建议优先改动的组件

### 1. `frontend/src/components/result-dashboard.tsx`

这是最关键的结果页组件，应优先重构。

建议方向：

- 重排主区布局
- 放大图像舞台
- 强化结论卡
- 弱化右侧次级工具栏
- 空结果状态视觉重写

### 2. `frontend/src/lib/result-utils.ts`

建议增强这些函数的表达能力：

- `getDetectionSummary()`
- `getPrimaryFinding()`
- `getResultNextStep()`

目标是从“基础描述”升级到“专业结果摘要”。

### 3. `frontend/src/components/home-shell.tsx`

建议统一 dashboard 首屏与结果页的叙事节奏，不要在 landing 和 dashboard 之间出现风格断层。

## 可直接落地的视觉准则

### 颜色

- 主背景：深黑蓝
- 主强调：电光蓝
- 辅强调：冰青色
- 成功状态：偏冷绿色
- 风险色：橙黄或裂缝红，仅用于病害高亮

### 圆角

- 大卡：24px 到 32px
- 中卡：18px 到 24px
- 小控件：10px 到 14px

### 阴影

- 以大面积柔和阴影为主
- 避免传统后台那种硬阴影
- 重要卡片可加淡蓝 halo

### 动效

- 卡片进入：淡入 + 轻微上移
- hover：边框亮度提升 + 阴影增强
- 避免频繁闪烁和强烈扫描动画

## 最值得先做的 6 件事

1. 放大中央识别图，建立主视觉。
2. 把“结果结论”改造成最强信息卡。
3. 用 landing 同款背景 glow 和玻璃面板统一视觉。
4. 重做按钮样式，提升产品感。
5. 将系统状态、筛选器降级，不再和主结论抢视线。
6. 重写无病害空状态，让它看起来像专业判断，而不是空页面。

## 一句话原则

这个页面不该只是“显示识别结果”，而应该让评委一眼感觉到：

`这是一套能把无人机桥梁图像转成可复核、可量化、可用于报告输出的专业 AI 工作台。`
