# 项目摘要

本项目面向无人机桥梁巡检场景，围绕桥梁病害识别、伪病害干扰抑制、结果量化分析与运维闭环管理，形成了一套完整的项目方案。报告既覆盖项目背景、系统创新、算法侧输入与项目管理，也重点呈现已经真实落地的软件系统实现部分。

前端采用 **Next.js 16 + React 19 + TypeScript + TailwindCSS 4** 构建多页面工作台，围绕桥梁资产、巡检批次、检测结果、历史档案、告警中心、复核记录和系统设置组织完整交互。后端采用 **FastAPI + Pydantic + SQLAlchemy + PostgreSQL** 搭建服务体系，通过统一接口组织单图推理、批量异步处理、结果查询、导出、诊断和运维管理。

模型接入层采用 **Registry + Runner + Adapter + Fusion** 的可插拔架构，支持主模型与专项模型协同工作。当前系统已实现主模型、渗水专项模型、裂缝专项模型的多模型融合，并支持增强后结果作为同一记录下的第二套正式识别结果。系统同时具备批次管理、任务重试、告警规则、审核闭环与历史追溯能力。

本文档按完整项目报告组织，其中第 5 章重点展开已经真实落地的软件系统实现，包括：

- 系统分层架构与关键模块职责

- 桥梁、批次、任务、结果的完整业务链路
- 多模型融合和增强结果在系统中的集成方式
- 关键页面、关键接口与关键实现机制

**关键词**：桥梁病害检测、多模型融合、图像增强、Next.js、FastAPI、异步任务、运维闭环

# 1 项目概要

## 1.1 项目背景

### 1.1.1 桥梁巡检现状

桥梁作为重要的交通基础设施，其数量庞大且分布广泛。以中国为例，截至2024年全国公路桥梁数量已超过100万座，其中大量桥梁建成时间较长，老化问题日益突出。传统的人工巡检方式存在以下问题：

**效率低下**：人工巡检需要检测人员携带设备攀爬桥梁，单座桥梁巡检耗时数小时至数天，无法满足大规模巡检需求。

**主观性强**：检测结果依赖于检测人员的经验和判断，不同检测人员可能给出差异较大的结论，缺乏客观统一的评判标准。

**安全隐患**：检测人员需要在高空、水下等危险环境作业，存在人身安全风险。

**数据孤岛**：巡检数据以纸质报告或独立文件形式存储，难以进行数据分析和历史对比，无法支持智能化决策。

### 1.1.2 无人机巡检趋势

近年来，无人机技术在基础设施巡检领域得到广泛应用。无人机巡检具有以下优势：

- **高效性**：单次飞行可完成大面积巡检，效率是人工巡检的10倍以上
- **安全性**：检测人员无需登高作业，降低安全风险
- **数据化**：可采集高分辨率图像、视频等多模态数据
- **可追溯**：每次巡检数据完整保存，支持历史对比分析

然而，无人机巡检产生的海量图像数据又带来了新的挑战——如何快速、准确地从图像中识别出病害信息？这正是本项目要解决的核心问题。

### 1.1.3 病害识别需求

桥梁病害种类繁多，常见的包括：

| 病害类型 | 英文名称 | 特征描述 |
|----------|----------|----------|
| 裂缝 | Crack | 桥梁表面出现的线性开裂，是最常见的病害类型 |
| 破损/剥落 | Spalling | 混凝土表层脱落，露出内部骨料或钢筋 |
| 钢筋外露 | Rebar Exposure | 混凝土保护层脱落导致钢筋暴露 |
| 渗水 | Seepage | 水分渗透痕迹，表面出现水渍、钙化物 |
| 孔洞 | Hole | 混凝土表面的空洞缺陷 |
| 梳齿缺陷 | Gear Defect | 伸缩缝梳齿板的损坏或缺失 |

传统人工识别方式难以在短时间内处理海量图像，且容易出现漏检、误检。因此，开发智能化的桥梁病害识别系统具有重要的现实意义。

## 1.2 项目目标

### 1.2.1 总体目标

本项目旨在开发一套完整的桥梁病害智能判读系统，实现从图像自动上传、病害智能识别到检测结果展示、报告生成的全流程自动化处理。

### 1.2.2 具体目标

**目标一：自动化病害识别**

- 支持6类常见桥梁病害的自动识别与分割
- 识别准确率达到行业领先水平
- 单张图像推理时间控制在合理范围内

**目标二：批量处理能力**

- 支持批量图像上传，单批次处理量可达千张
- 异步任务队列机制，支持大规模并发处理
- 实时任务进度监控

**目标三：运维闭环管理**

- 检测结果审核与标注
- 疑似病害自动告警
- 历史数据查询与对比
- 告警处理与闭环管理

**目标四：智能化诊断**

- 基于检测结果自动生成病害诊断报告
- 提供养护建议参考

# 2 系统架构设计

## 2.1 设计目标与边界

本系统的软件架构围绕“桥梁巡检结果的稳定入库、可追溯查询与可视化复核”展开，重点不是构建一个通用 AI 平台，而是完成以下四类能力的闭环：

1. 桥梁资产与巡检批次管理  
2. 图片上传、任务调度与结果落库  
3. 单图详情、历史档案、复核与告警联动  
4. 多模型融合、增强结果与诊断结果的系统承接

本章描述的是系统已经落地的软件结构，而不是抽象产品蓝图。前端、后端、模型运行时与数据库均以当前实现为基础进行说明。

## 2.4 后端服务协作结构

后端采用 `FastAPI + Pydantic + SQLAlchemy`。整体风格不是“胖路由”，而是“接口入口 -> 业务服务 -> 读写与执行子模块 -> 数据模型与推理引擎”。

```mermaid
flowchart TB
    V1["运营工作台接口"] --> BS["BatchService (门面)"]
    V1 --> TS["TaskService (门面)"]
    L1["单图结果接口"] --> PS["PredictService"]
    L1 --> RS["ResultService"]

    BS --> BQ["Batch Query Services"]
    BS --> BW["Batch Write / Validation / Ingest"]
    BS --> BAR["Alert / Review Service"]

    TS --> TE["Task Execution Service"]
    TS --> TR["Task Retry Service"]
    TS --> TA["Task Alert Service"]

    PS --> RM["RunnerManager"]
    RM --> FR["FusionRunner"]
    RM --> ER["External Runtimes"]
    RS --> DB[(PostgreSQL)]
    BS --> DB
    TS --> DB
    PS --> DB
```

这种分层的目的，是把“接口承接”“业务编排”“查询优化”“模型执行”拆开，避免所有逻辑堆在一个 service 文件中。

## 2.5 运行时协作与结果落库

批量巡检链路不是单次请求完成，而是“批次创建 -> 素材入库 -> 任务队列 -> worker 执行 -> 结果写回”的异步链路。

```mermaid
sequenceDiagram
    participant UI as 前端工作台
    participant API as FastAPI
    participant BS as BatchService
    participant TS as TaskService
    participant RT as RunnerManager/FusionRunner
    participant DB as PostgreSQL
    participant FS as 文件产物目录

    UI->>API: POST /api/v1/batches
    API->>BS: create_batch(payload)
    BS->>DB: 写入 inspection_batches
    UI->>API: POST /api/v1/batches/{id}/items
    API->>BS: ingest_items(files, relative_paths, policy)
    BS->>DB: 写入 media_assets / batch_items / inference_tasks
    TS->>DB: process_next_queued_task()
    TS->>RT: resolve model + run inference
    RT->>FS: 生成 overlay / enhanced / json 产物
    TS->>DB: 写入 inference_results / detections / batch 聚合状态
    UI->>API: GET stats/items/result
    API->>DB: 查询并返回结构化结果
```

这条链路也是当前系统最核心、最有工程价值的部分：它把模型结果变成了可管理、可追溯、可回放的业务资产。

## 2.6 多模型与增强结果的承接方式

当前系统不只支持单模型推理，还支持：

- 主模型推理
- 渗水专项模型覆盖
- 裂缝专项模型覆盖
- 多模型 fusion 输出
- 增强后图片与增强后结果的第二套结果承接

```mermaid
flowchart LR
    I["原始图像"] --> M["主模型"]
    I --> W["渗水专项模型"]
    I --> C["裂缝专项模型"]
    I --> E["增强处理"]
    E --> ER["增强结果推理"]

    M --> F["Fusion 结果整合"]
    W --> F
    C --> F
    ER --> R2["secondary_result"]
    F --> R1["primary_result"]

    R1 --> D["详情页 / 历史页 / 检索页"]
    R2 --> D
```

这里的关键设计不是“模型越多越好”，而是把多模型输出统一归一成同一套 `PredictResponse`/`Detection` 结构，再由前端页面以统一方式消费。

# 3 技术实现详解

本章聚焦系统组已完成的软件实现，不讨论模型训练论文式细节，而说明当前系统如何支撑桥梁资产、巡检批次、检测结果、历史档案、告警复核与增强结果等核心链路。

---

## 3.1 前端技术栈与实现

### 3.1.1 技术选型

| 技术 | 角色 | 在本项目中的用途 |
|------|------|------------------|
| Next.js 16 | 应用框架 | App Router、多页面组织、构建产物 |
| React 19 | 组件模型 | 页面组合、状态驱动 UI |
| TypeScript | 类型系统 | 接口消费、组件约束、重构保护 |
| TailwindCSS 4 | 样式系统 | 统一视觉与快速布局 |
| Framer Motion | 动效层 | 页面过渡、面板展开、状态反馈 |
| Vitest | 单元测试 | 工具函数、关键页面回归 |
| Playwright | E2E | real-mode 浏览器链路验证 |

### 3.1.2 前端分层结构

前端不是把所有逻辑都堆在页面文件中，而是围绕“页面入口、页面组织、状态与动作、业务区块、接口访问”做层次划分。

```mermaid
flowchart TB
    A["页面入口层"] --> B["页面组织层"]
    B --> C["状态与动作层"]
    C --> D["业务区块层"]
    C --> E["领域接口层"]
```

**图 3-1 前端分层结构**

这套结构的重点是：页面本身只负责表达界面，状态变化和业务动作被单独组织，局部区域也被拆成可复用区块。因此在迭代桥梁、批次、历史和详情这些页面时，不需要重新把请求、状态和视图全部堆回一个组件中。

若需要将这些层次与真实目录对应起来，可参见附录 E 的前端页面入口层、页面组件层和客户端与状态层。

表 3-1 给出前端分层与实现方式的对应关系。

| 层级 | 设计职责 | 实现方式 |
|------|----------|----------|
| 页面入口层 | 负责页面路径与页面装配入口 | 以页面路由文件作为每个业务页面的入口 |
| 页面布局层 | 负责页面主布局、标题区和区域装配 | 以桥梁页、批次页、历史页、详情页为代表的页面容器 |
| 页面控制层 | 负责把页面状态、动作和局部区块组合成完整界面 | 由页面控制逻辑完成数据与界面装配 |
| 状态管理层 | 负责 URL 同步、轮询、数据加载与派生状态 | 通过状态模块维护当前桥梁、批次、列表和详情上下文 |
| 动作处理层 | 负责上传、创建、删除、重试、状态更新 | 通过动作模块统一承接写操作与刷新逻辑 |
| 视图区块层 | 负责批次面板、记录清单、告警列表、详情区渲染 | 把复杂页面拆成多个职责明确的可复用区块 |
| 接口访问层 | 负责按业务域访问后端 API | 按桥梁、批次、告警、复核、设置等领域拆分请求封装 |

### 3.1.3 真实项目结构

当前前端实现可以归纳为四类模块：

| 模块类型 | 作用 |
|------|------|
| 路由模块 | 负责页面入口与路由分发 |
| 页面组件模块 | 负责桥梁、批次、历史、告警、复核、详情等业务界面 |
| 客户端与工具模块 | 负责 API 请求、类型、工具函数与动画能力 |
| 回归验证模块 | 负责 Vitest 单测与 real-mode 浏览器回归 |

### 3.1.4 页面结构与状态组织

本项目对复杂页面采用统一的组织思路：

1. 页面主结构只负责布局。  
2. 页面状态负责 URL、轮询、数据加载和派生状态。  
3. 页面动作负责创建、上传、删除和状态更新。  
4. 局部区块负责把每个功能区域独立渲染出来。  

以批次工作台为例：

```mermaid
graph LR
    A["工作台主结构"] --> B["页面控制逻辑"]
    B --> C["状态组织"]
    B --> D["动作处理"]
    A --> E["导航区"]
    A --> F["主体区"]
    A --> G["总览区"]
```

**图 3-2 批次工作台的壳组件与控制结构**

这样设计的直接价值是：

- 大页面文件可以持续瘦身  
- 状态和副作用更容易测试  
- 后续改桥梁、批次、历史、告警时，不需要重写整页结构

### 3.1.5 前端页面设计

前端页面设计不是简单地把不同页面放到不同路由下，而是围绕“资产视角、批次视角、记录视角、运维视角”做职责划分。这样设计的原因是，不同页面承接的是不同粒度的业务上下文：

- 桥梁页负责桥梁资产与桥梁级批次摘要；
- 批次工作台负责当前桥梁下的批次处理和素材清单；
- 单图详情页负责单条记录的识别结果、增强结果、诊断与导出；
- 历史档案页负责历史回查；
- 告警/复核/设置页负责运维闭环。

表 3-2 给出主要页面的设计定位。

| 页面 | 主要用户目标 | 关键状态 | 关键跳转 |
|------|--------------|----------|----------|
| 桥梁资产页 | 浏览桥梁、进入桥梁详情、创建桥梁 | 桥梁列表、筛选条件 | 进入桥梁详情、进入批次中心 |
| 桥梁详情页 | 查看桥梁最近批次、告警和风险摘要 | 当前桥梁、最近批次、开放告警 | 返回桥梁页、进入批次中心 |
| 批次工作台 | 创建批次、上传图片、查看任务状态 | 当前桥梁、当前批次、上传状态、轮询状态 | 进入单图详情、返回桥梁页 |
| 单图详情页 | 查看识别结果、增强结果、诊断与导出 | 视图模式、结果来源、检测聚焦项 | 返回工作台、进入历史详情 |
| 历史档案页 | 按批次回查记录并打开历史详情 | 当前批次、记录筛选、历史上下文 | 打开历史详情、返回批次工作台 |
| 告警页 | 查看告警、更新状态、批量处理 | 过滤条件、SLA 视图、批量选择 | 打开详情、进入批次相关页面 |
| 复核页 | 查看待复核项并记录人工复核动作 | 复核筛选、排序、状态 | 打开详情、回到复核列表 |

### 3.1.6 前端接口客户端分层

前端不再使用一个超大 API 文件，而是按运营域拆分。桥梁、批次、检测、告警、复核、设置等请求分别组织，再通过统一导出层向页面提供稳定入口。这样兼顾了两点：

1. 页面侧导入路径稳定  
2. 内部实现按业务域拆开，降低耦合

前端在调用批次创建接口时，已经把“页面状态”和“请求结构”明确分开。页面层只面向桥梁、批次、来源、备注等业务语义字段编程，而字段命名转换和请求封装留在客户端层完成。

表 3-2A 给出这种映射方式的典型例子。

| 页面语义字段 | 接口字段 | 说明 |
|--------------|----------|------|
| `bridgeId` | `bridge_id` | 当前桥梁上下文 |
| `sourceType` | `source_type` | 素材来源类型 |
| `expectedItemCount` | `expected_item_count` | 预期上传数量 |
| `createdBy` | `created_by` | 创建人 |
| `inspectionLabel` | `inspection_label` | 批次说明或标签 |
| `enhancementMode` | `enhancement_mode` | 增强策略 |

这种封装的作用是：页面组件只面向前端语义字段编程，而接口字段适配留在客户端层完成。

附录 E 的前端客户端与状态层部分补充了这些模块在工程中的组织位置。

### 3.1.7 前端回归保障

前端测试分为两类：

1. **Vitest**  
   用于组件、工具函数、客户端层回归。
2. **Playwright real-mode**  
   直接连接 `bdi run` 启动的真实前后端链路。

real-mode E2E 当前覆盖的主路径包括：

- 批次创建与上传
- 上传校验
- 历史档案
- 告警页
- 复核页
- 设置页
- 桥梁详情页
- 桥梁删除级联

这意味着前端不仅“能渲染”，而且已经有浏览器级的真实链路验证。

## 3.2 后端技术栈与实现

### 3.2.1 技术选型

| 技术 | 角色 | 在本项目中的用途 |
|------|------|------------------|
| FastAPI | Web 框架 | 路由组织、请求处理、自动文档 |
| Pydantic | 数据模型 | 请求校验、响应序列化、类型约束 |
| SQLAlchemy 2 | ORM | 数据读写、关系映射 |
| PostgreSQL | 主数据库 | 结构化业务数据持久化 |
| Uvicorn | ASGI 服务 | 本地开发、演示启动 |

### 3.2.2 后端服务架构

后端当前并不是“单文件式服务”，而是基于门面服务继续拆分：

```mermaid
graph TB
    A["接口入口"] --> B["业务服务门面"]
    B --> C["批次处理模块"]
    B --> D["任务处理模块"]
    B --> E["结果与诊断模块"]
    C --> F["查询 / 写入 / 校验 / 告警复核"]
    D --> G["执行 / 重试 / 增强 / 任务告警"]
```

**图 3-4 后端门面服务与领域子服务拆分**

这种设计的目的不是“文件拆分得更碎”，而是为了让：

- 路由层保持薄
- 门面服务保持稳定
- 读、写、校验、告警、增强等逻辑可以独立演进

表 3-3 给出当前后端核心服务的职责划分。

| 服务层级 | 代表模块 | 主要职责 |
|----------|----------|----------|
| 路由层 | 路由模块 | 参数接收、响应类型声明、调用业务入口 |
| 业务入口层 | Predict、Batch、Task、Result、LLM 服务 | 维持对外稳定接口，统一业务入口 |
| 批次处理层 | 批次校验、入库、写操作、查询、告警与复核联动模块 | 承担桥梁和批次相关的读写编排 |
| 任务处理层 | 任务执行、失败重试、增强补算、任务告警模块 | 承担异步任务与增强链路 |
| 引擎适配层 | 模型注册、运行时装配、多模型融合模块 | 把模型能力统一为同一套结果协议 |

后端路由与服务之间的关系也保持一致：请求只在接口层完成参数接收，真正的业务编排交给门面服务和领域子模块处理。

若需要继续追踪这些门面服务和子服务的工程边界，可结合附录 E 的后端服务导航阅读。

### 3.2.3 Legacy API 与 V1 API

后端接口分成两条主线：

| 接口组 | 用途 |
|--------|------|
| Legacy API | 单图推理、结果读取、增强、诊断、导出 |
| V1 API | 桥梁、批次、任务、检测、复核、告警、设置 |

这种分离带来的好处是：

1. 单图实验页和企业工作台链路互不干扰  
2. 批量运维能力可以独立演进  
3. 前端可以根据页面语义选择不同接口组

### 3.2.4 请求追踪与错误处理

后端已经统一接入：

- `X-Request-ID`
- `X-Process-Time-Ms`
- 统一错误结构

请求进入后的处理过程可以表示为：

```mermaid
sequenceDiagram
    participant Client as 前端
    participant API as FastAPI
    participant Middleware as Trace Middleware
    participant Service as Service Layer
    participant DB as DB/Storage

    Client->>API: 发起请求
    API->>Middleware: 分配 Request ID
    Middleware->>Service: 调用业务逻辑
    Service->>DB: 读写数据
    DB-->>Service: 返回结果
    Service-->>Middleware: 业务响应 / AppError
    Middleware-->>Client: 标准响应 + Request ID + Process Time
```

**图 3-5 请求追踪与统一错误处理**

这使得页面错误与后端日志之间可以直接关联，而不是只能靠人工猜测。

1. 数据库负责结构化关系与状态管理  
2. 文件系统负责图片与结果媒体产物，不把大体积文件直接塞进数据库

### 3.3.2 业务主对象

当前系统围绕四类核心对象组织：

| 对象 | 作用 |
|------|------|
| Bridge | 长期资产上下文 |
| InspectionBatch | 某座桥的一次巡检会话 |
| BatchItem | 批次中的单张图片记录 |
| InferenceTask / InferenceResult | 推理执行与结果落库 |

这也是为什么前端工作台和历史页都围绕“桥梁 -> 批次 -> 记录”组织。

表 3-4 给出核心数据表的设计定位。

| 表名 | 角色 | 关键字段 | 说明 |
|------|------|----------|------|
| `bridges` | 桥梁资产主表 | `id`、`bridge_code`、`name`、`region`、`status` | 表示长期存在的桥梁资产 |
| `inspection_batches` | 巡检批次主表 | `id`、`bridge_id`、`batch_code`、`status`、`source_type` | 表示某座桥的一次巡检会话 |
| `media_assets` | 媒体资源表 | `id`、`storage_path`、`mime_type`、`original_name` | 保存原图与产物的媒体元信息 |
| `batch_items` | 批次记录表 | `id`、`batch_id`、`media_asset_id`、`latest_task_id` | 表示批次中的单张图片记录 |
| `inference_tasks` | 任务执行表 | `id`、`batch_item_id`、`status`、`runtime_payload`、`retry_count` | 表示异步推理与增强处理任务 |
| `inference_results` | 结果快照表 | `id`、`task_id`、`model_version`、`secondary_result` | 保存识别结果、增强结果和诊断数据 |
| `detections` | 病害明细表 | `id`、`result_id`、`category`、`confidence`、`bbox`、`mask` | 保存结构化病害检测结果 |
| `alert_events` | 告警事件表 | `id`、`batch_item_id`、`status`、`severity`、`rule_code` | 承接自动触发的运维风险事件 |
| `review_records` | 复核记录表 | `id`、`batch_item_id`、`action`、`reviewed_by` | 保存人工复核动作与意见 |

### 3.3.3 查询优化策略

项目后期已经开始对数据库读取做结构优化，典型措施包括：

- 将批次读域拆成查询服务
- 减少 `N+1` 查询
- 批次素材和详情通过 join / 聚合一次带出必要字段
- 总览页改成 KPI 优先，不再首屏全量拉取明细

这说明数据库设计不只是表结构设计，还包括读路径的持续优化。

## 3.4 工程实践与质量保障

### 3.4.1 契约与类型一致性

项目已经做了三层一致性约束：

1. 后端以 Pydantic 模型定义协议  
2. OpenAPI 生成前端类型  
3. 前端只在展示层做最少量兼容处理

这使得前后端协议不再靠口头同步。

### 3.4.2 组件与服务持续拆分

在项目推进过程中，若某个页面、服务或客户端模块体量过大，就会继续按领域拆分。这一点已经在以下模块中落地：

- 任务处理服务
- 批次管理服务
- 前端接口访问层
- 批次工作台
- 历史档案页
- 告警中心页
- 结果详情页

这类拆分带来的收益，不只是“文件更短”，而是：

- 回归范围变小
- Coding agent 更容易准确修改
- Review 成本下降

## 4.1 单图推理与结果展示

### 4.1.1 功能定位

单图推理模块承担两个角色：

1. 作为最直接的模型能力展示入口  
2. 作为结果详情页、增强结果和导出能力的基础入口

### 4.1.2 用户链路

```mermaid
flowchart LR
```

## 5.2 软件接口规范

### 5.2.1 接口分层

后端接口分为两组：

1. `Legacy API`  
   面向单图推理、结果回看和增强结果。
2. `V1 API`  
   面向桥梁、批次、任务、告警、复核和运营工作台。

```mermaid
graph LR
    A["单图结果接口"] --> A1["推理 / 结果 / 增强 / 诊断"]
    B["运营工作台接口"] --> B1["桥梁 / 批次 / 任务"]
    B --> B2["检测 / 复核 / 告警 / 设置"]
```

**图 5-3 接口分层结构**

### 5.2.2 接口组织原则

本项目接口设计遵循三条原则：

1. **单图链路与批量链路分开**  
   单图使用 `Legacy API`，批量和运维使用 `V1 API`。
2. **前后端类型以 OpenAPI 契约为准**  
   前端通过生成类型消费后端协议，减少手工维护偏差。
3. **错误返回统一结构**  
   所有核心接口都接入统一错误响应与请求追踪。

统一错误结构：

```json
{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "relative_paths 与 files 数量不一致",
    "details": {}
  }
}
```

同时，每个请求都会带：

- `X-Request-ID`
- `X-Process-Time-Ms`

这保证了页面报错可以直接回溯到后端日志。

### 5.2.3 实际接口代码组织

在当前系统中，Legacy API 与 V1 API 分别承担“单图实验链路”和“企业工作台链路”。接口组织的核心原则不是堆更多端点，而是让路由层足够薄、服务层足够稳定、客户端调用足够统一。

例如，批次创建与批次图片上传都遵循同一套组织方式：前端先把页面语义参数整理为稳定的请求对象，再由接口入口交给批次服务统一处理。这样，页面层不需要直接理解后端字段细节，接口层也不需要承担完整的业务规则。

如需继续从接口入口追踪到服务层，可参见附录 E 的接口入口层与业务服务层说明。

图 5-6 反映的是后端服务从“单体大服务”演进到“门面服务 + 领域子服务”的过程。门面层负责向路由层提供稳定入口，Batch 子服务组负责批次校验、入库、写操作、查询、告警与复核联动，Task 子服务组负责异步执行、失败重试、增强补算和任务告警。这种设计既保留了 API 层调用习惯，又降低了单个服务文件持续膨胀的风险。

门面层负责向接口层提供稳定入口，具体的校验、入库、查询、告警与复核处理则下沉到各自的业务模块中。这种做法既保留了对外接口的稳定性，也避免单个服务文件继续膨胀。

这些门面与子服务在工程中的主要分布，可在附录 E 的后端服务层中继续查看。

### 5.4.3 模型引擎设计

模型引擎的关键目标不是简单加载一个 `.pt` 文件，而是支持：

- 主模型版本管理  
- 多专项模型覆盖  
- 融合输出统一为一套结果协议  
- 运行时与权重路径解耦

```mermaid
flowchart LR
    A["配置与模型注册"] --> B["RunnerManager"]
    B --> C["主模型 Runner"]
    B --> D["专项模型 Runner"]
    C --> E["FusionRunner"]
    D --> E
    E --> F["统一结果对象"]
```

**图 5-7 模型引擎与多专项融合结构**

图 5-7 展示的是模型引擎的核心设计：配置层与注册层决定可用模型版本，`RunnerManager` 负责装配主模型和专项模型 runner，`FusionRunner` 负责把主模型结果与渗水、裂缝等专项模型结果融合成统一结果对象。这样设计的目标不是追求单一模型入口，而是让主模型与专项模型能够在统一协议下协同工作，并稳定回传到业务链路。

这套设计使当前系统能够同时接入：

- 主模型 `main-latest-mask-v1`
- 渗水专项模型
- 裂缝专项模型
- 多专项融合版本 `fusion-main-water-crack-mask-v1`

融合阶段会根据类别和 IoU 去重规则完成专项覆盖与结果合并。后端真实颜色定义也在融合层统一输出，保证前端列表、框线、overlay 图颜色一致。

### 5.4.5 接口入口与服务协作

这一层的关键不是某一个具体函数，而是接口入口和业务服务之间的协作边界已经稳定：

- 接口入口负责参数接收、响应组织和错误回传；

- 模型引擎负责推理执行、融合和结果统一。

### 5.4.6 持久化设计

数据库与文件产物是并行存储的：

| 类型 | 位置 | 内容 |
|------|------|------|
| 结构化业务数据 | PostgreSQL | Bridge、Batch、Task、Result、Detection、Alert、Review |
| 上传原图与结果媒体 | artifact 存储目录 | 原图、overlay、增强图、增强 overlay |
| 模型权重 | `backend/models` | 主模型、专项模型 |
| 外部运行时 | `backend/external_runtimes` | 多运行时模型推理环境 |

这样做的好处是：

1. 数据库负责结构化管理  
2. 文件系统负责承接体积更大的图像与结果产物  
3. 结果详情页和历史页可以直接通过 `image / overlay / enhanced / enhanced-overlay` 路由取资源

### A.2.3 单图推理

**POST `/predict`**

这是单图实验页和传统结果链路的入口。请求采用 `multipart/form-data`。

核心参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `file` | file | 原始图片 |
| `confidence` | float | 置信度阈值 |
| `iou` | float | IoU 阈值 |
| `inference_mode` | string | 推理模式 |
| `model_version` | string | 指定模型版本，可为空表示当前 active |
| `return_overlay` | bool | 是否返回 overlay 链接 |
| `pixels_per_mm` | float | 像素与物理尺寸换算系数 |

```python
@router.post("/predict", response_model=PredictResponse)
async def predict(
    request: Request,
    file: Annotated[UploadFile, File(...)],
    confidence: Annotated[float, Form()] = 0.25,
    iou: Annotated[float, Form()] = 0.45,
    inference_mode: Annotated[str, Form()] = "direct",
    model_version: Annotated[Optional[str], Form()] = None,
    return_overlay: Annotated[bool, Form()] = False,
    pixels_per_mm: Annotated[float, Form()] = 10.0,
) -> PredictResponse:
    ...
```

响应结构核心字段：

| 字段 | 说明 |
|------|------|
| `image_id` | 结果主键 |
| `result_variant` | `original` 或 `enhanced` |
| `inference_ms` | 推理耗时 |
| `model_version` | 实际模型版本 |
| `detections` | 检测结果数组 |
| `artifacts` | 原图、overlay、json 等链接 |
| `secondary_result` | 增强后的第二套结果 |

# 附录 B：数据库设计

## B.1 设计说明

系统数据库采用 `PostgreSQL + SQLAlchemy ORM`。数据库并不只是保存“检测结果”，而是同时承担以下职责：

1. 保存桥梁、批次、任务等业务对象  
2. 保存单图推理与批量推理的结构化结果  
3. 保存告警、复核、配置、审计等运营数据  
4. 与文件产物目录共同完成“结构化数据 + 媒体资源”的双轨持久化

实际数据模型定义位于：

- 桥梁资产模型
- 巡检批次模型
- 批次记录模型
- 推理任务模型
- 结果与检测模型
- 告警、复核、配置与审计模型

数据库设计并不是“单张图片结果表”的扩展，而是围绕桥梁巡检闭环完整建模。下面这段 ORM 代码可以说明桥梁与批次的一对多关系是如何在实现层表达的：

```python
class Bridge(Base):
    __tablename__ = "bridges"

    id = mapped_column(String, primary_key=True)
    bridge_code = mapped_column(String, unique=True, nullable=False)
    bridge_name = mapped_column(String, nullable=False)
    status = mapped_column(String, nullable=False, default="active")

    batches = relationship("InspectionBatch", back_populates="bridge", cascade="all, delete-orphan")


class InspectionBatch(Base):
    __tablename__ = "inspection_batches"

    id = mapped_column(String, primary_key=True)
    bridge_id = mapped_column(ForeignKey("bridges.id"), nullable=False)
    batch_code = mapped_column(String, unique=True, nullable=False)
    status = mapped_column(String, nullable=False, default="created")

    bridge = relationship("Bridge", back_populates="batches")
```

## D.4 图尺寸统一原则

Mermaid 在 Typora 中很难做到“每张图强制同宽”，因此统一图尺寸的主要方法不是继续调样式，而是控制图本身：

2. 节点控制在 6 到 9 个左右  
3. 节点文字尽量控制在 6 到 14 个字  
4. 同一章中尽量统一方向：

## E.2 从业务主链路理解项目结构

| 接口与服务层 | `backend/app/api/`、`backend/app/services/` | 负责参数接收、业务编排、查询与写入协作 |
| 数据与引擎层 | `backend/app/db/`、`backend/app/models/`、`backend/app/adapters/` | 负责 ORM、协议模型、模型运行时与融合逻辑 |

```python
@router.post("/batches", response_model=BatchCreateResponse, status_code=status.HTTP_201_CREATED)
async def create_batch(request: Request, payload: BatchCreateRequest) -> BatchCreateResponse:
    return request.app.state.batch_service.create_batch(payload)
```

这类写法说明：真正的业务编排不放在路由层，而是交给门面服务处理。

```python
def process_next_queued_task(self) -> TaskProcessResponse:

```

这种门面结构的意义在于：对外接口保持稳定，而执行、重试、增强和告警逻辑可以继续独立演进。

这类配置使主模型、渗水专项模型和裂缝专项模型可以在统一协议下协同工作。
