# 模型对齐说明

## 这份文档是做什么的

这份文档是给我自己看的。

目的不是讲完整的深度学习理论，而是帮助我在和算法同学沟通时，明确知道：

- 现在还缺哪些关键信息
- 为什么这些信息必须确认
- 不同类型的模型文件到底代表什么
- 当前项目为什么会出现“能检测但没有掩膜”的情况
- 后续怎么把“通用模型”和“渗水专项模型”接进系统

## 当前已经确认的事实

我目前手里的模型相关内容有三类：

1. `backend/models/best.pt`
2. 桌面上的 `best.pt`
3. `PRNet-main`

目前已经确认：

- `backend/models/best.pt` 是老版本通用模型权重
- 桌面上的 `best.pt` 是渗水专项模型权重
- `PRNet-main` 当前是代码工程目录，不是已经可直接推理的最终权重

另外已经确认一个重要事实：

- 目前拿到的两个 `best.pt` 都是 `detect` 模型，不是 `segment` 模型

这意味着它们可以输出：

- 病害类别
- 置信度
- 检测框

但不能输出：

- 实例掩膜 `mask`
- 基于真实掩膜的面积、长度、宽度等精细量化结果

## 我是怎么判断它是 `detect` 模型的

这部分很重要，因为这不是“猜的”，而是根据权重内部信息和当前系统行为一起判断出来的。

我这次的判断依据主要有三层。

### 第一层：直接读取权重里的训练信息

我用项目里的真实模型虚拟环境去读取了权重内部保存的元信息。

从权重里读到的关键信息包括：

- 桌面渗水模型的 `train_args.task = detect`
- 老通用模型的 `train_args.task = detect`
- 桌面渗水模型的 `model_class = DetectionModel`

这几个信息本身就已经足够说明问题。

因为如果是分割模型，通常至少应该看到：

- `task = segment`
或
- 模型结构对应分割任务，而不是 `DetectionModel`

也就是说，权重在训练阶段就已经被定义成“检测任务”了，不是“实例分割任务”。

### 第二层：当前后端适配逻辑没有拿到 `mask`

当前后端在适配 Ultralytics 输出时，逻辑大致是这样的：

- 先读取检测框
- 再检查 `result.masks` 是否存在
- 如果 `result.masks` 存在，才会生成系统里的 `mask`

如果模型本身不是分割模型，那么推理结果里通常不会有有效的 `result.masks`。

这样一来，后端最终返回的就只有：

- 类别
- 置信度
- 检测框

而不会有真实掩膜。

所以“系统没有生成掩膜”这件事，不只是现象，也反过来印证了权重类型和任务类型不匹配。

### 第三层：权重训练信息和实际现象是互相吻合的

如果只是看“没有掩膜”这个现象，还不能百分百下结论。

因为理论上也可能是：

- 后处理写错了
- 掩膜字段没适配好
- 推理代码漏取了 `masks`

但是这次不是这样。

因为我已经同时拿到了两类证据：

1. 权重内部训练信息明确显示是 `detect`
2. 当前系统运行现象也确实没有拿到 `mask`

这两者是完全一致的。

所以结论不是“后端偶然没出掩膜”，而是：

**当前拿到的权重本身就是检测模型，因此它从能力上就不提供实例分割掩膜。**

## 我是怎么一步步发现这个问题的

这部分是完整的问题发现过程。

### 第一步：先确认你手里到底有哪些文件

先看了你机器上的实际文件，确认到：

- 项目里有一个老的 `backend/models/best.pt`
- 桌面上还有一个新的 `best.pt`
- 桌面上有一个 `PRNet-main` 目录

这一步的目的是先把“文件层面”的情况搞清楚，避免把不同来源的文件混为一谈。

### 第二步：确认 `PRNet-main` 里有没有现成权重

检查后发现：

- `PRNet-main` 是代码工程
- 目录里没有现成可直接推理的本地权重文件
- README 里提到 checkpoint 需要额外下载

这一步发现了第一个关键问题：

**你现在不能把 `PRNet-main` 直接当成一个现成模型接进系统。**

### 第三步：对比几个 `best.pt` 的体积和来源

我对比后发现：

- 桌面新 `best.pt` 只有约 `5.2MB`
- 桌面 `数据集/best.pt` 约 `99MB`
- 项目里的 `backend/models/best.pt` 也是约 `99MB`

后两个大小完全一致，基本可以判断项目里现有的老通用模型和桌面 `数据集` 里的模型是同一个来源。

而 `5.2MB` 的那个显然是另一个新模型。

### 第四步：读取权重内部元信息

这是关键一步。

读取之后发现：

- 新的渗水模型训练任务是 `detect`
- 老的通用模型训练任务也是 `detect`
- 新的渗水模型类别数 `nc = 1`
- 新的渗水模型训练数据指向 `seepage.yaml`

这说明：

- 新模型是单类渗水检测模型
- 老模型是通用检测模型
- 两者都不是实例分割模型

### 第五步：把权重信息和系统现象对应起来

最后再结合当前项目的输出行为去验证：

- 系统能拿到类别和框
- 但拿不到真实 `mask`

这和前面读到的 `detect` 权重信息完全一致。

所以问题真正的根因被定位出来了：

**不是代码没写掩膜，而是同学给到的这批权重，本身就不是分割权重。**

## 为什么现在没有生成掩膜

原因不是后端没写好，而是权重本身的任务类型不对。

在 YOLO 体系里，常见任务类型至少有两种：

1. `detect`
2. `segment`

### `detect` 是什么

`detect` 是目标检测。

它通常只能告诉系统：

- 图里有什么病害
- 病害大概在哪个矩形框里

输出更像这样：

```text
类别 = 渗水
置信度 = 0.91
框 = x1, y1, x2, y2
```

### `segment` 是什么

`segment` 是实例分割。

它除了知道“哪里有病害”，还会给出病害轮廓，也就是更细的像素级区域或多边形区域。

输出更像这样：

```text
类别 = 渗水
置信度 = 0.91
框 = x1, y1, x2, y2
mask = 一组轮廓点
```

### 为什么分割更重要

如果系统后面要做这些能力：

- 掩膜展示
- 病害面积计算
- 裂缝长度估算
- 更精细的可视化结果

那就必须依赖 `segment` 权重，而不是 `detect` 权重。

因为面积和轮廓这些信息，本质上来自 `mask`，不是来自矩形框。

一个框只能告诉我们“病害大概在这里”，不能准确告诉我们“病害真实占了哪些像素”。

## 我还需要和算法同学对齐什么

下面这些信息，不是“可有可无”，而是模型能不能稳定接进产品系统的关键。

## 一、先确认每个权重到底是什么

### 要确认什么

每一个权重文件都要确认以下字段：

- 这个权重的用途是什么
- 这是通用模型还是专项模型
- 训练任务是 `detect` 还是 `segment`
- 识别哪些类别
- 类别顺序是什么
- 输入尺寸是多少
- 是否基于官方 `ultralytics` 直接训练
- 是否依赖自定义代码结构

### 为什么必须确认

因为文件名叫 `best.pt` 没有任何实际信息量。

如果不确认这些内容，就会出现下面这些问题：

- 同名权重被覆盖
- 前端显示类别不对
- 后端按分割处理检测模型，结果当然没有 `mask`
- 新模型加载失败，但不知道是路径问题还是结构不兼容
- 模型切换后效果变差，却不知道切换的是哪一个版本

### 我应该怎么记录

每个权重至少要有一份“模型卡片”，内容如下：

```text
模型名称：
模型版本：
用途：
任务类型：detect / segment
类别列表：
输入尺寸：
训练基座：
是否依赖 PRNet-main 自定义代码：
是否可直接用 ultralytics 加载：
权重路径：
备注：
```

## 二、确认 `PRNet-main` 到底给的是代码还是能跑的模型

### 当前情况

当前已经确认：

- `PRNet-main` 是代码仓库
- 它本身不等于最终权重
- 目录里暂时没有本地现成的 `.pt` / `.onnx` / `.engine` 权重可直接接入

### 为什么这件事重要

因为“给了一个模型工程”不等于“给了一个能在我这边直接推理的模型”。

产品系统真正需要的是：

- 一个最终权重文件
- 一个可加载方式
- 一个类别定义

而不是只给一个论文复现仓库。

### 我要和同学确认什么

需要直接问清楚：

1. `PRNet-main` 对应的最终权重在哪里
2. 这个权重是 `.pt`、`.onnx` 还是别的格式
3. 这个权重能不能被当前官方 `ultralytics` 直接加载
4. 如果不能，是否必须依赖 `PRNet-main` 里的自定义模块
5. 如果依赖自定义模块，是否有固定运行环境

### 技术解释

这件事的本质是“推理运行时兼容性”。

举个最简单的例子：

- 如果同学给的是官方 YOLO 兼容的 `.pt`
  那当前系统大概率可以直接加载
- 如果同学给的是改过网络结构的 `.pt`
  那当前环境里的官方 `ultralytics` 很可能加载失败

因为权重文件里记录的不只是参数，还可能依赖特定的模型结构定义。

## 三、确认每个模型的类别定义

### 要确认什么

我要明确知道每个模型到底识别哪些类。

比如：

- 通用模型识别 6 类
- 渗水专项模型只识别 1 类渗水

还要知道类别的标准名称和索引顺序。

### 为什么必须确认

如果类别定义不清楚，系统会出现这些问题：

- 类别映射错误
- 模型输出“water seepage”，系统却按别的病害显示
- 双模型融合时无法判断谁负责哪一类
- 导出结果和报告内容不可信

### 技术解释

模型输出的原始类别通常只是数字索引，例如：

```text
0 = seepage
1 = crack
2 = spalling
```

如果系统不知道这个映射关系，就无法正确显示中文病害名。

所以必须确认：

- 类别索引
- 英文原始名称
- 对应的标准中文类别

## 四、确认模型输入尺寸和推理设置

### 要确认什么

- 训练时图像尺寸是多少
- 推理推荐尺寸是多少
- 置信度阈值大概多少合适
- IoU 阈值大概多少合适
- 是否建议切片推理

### 为什么必须确认

因为输入尺寸和阈值会直接影响效果。

如果训练时是 `960` 或 `1024`，结果你推理时随便改成别的设置，可能会明显掉点。

特别是桥梁病害里很多目标比较细、小、形状复杂，输入尺寸差异可能会直接影响检测稳定性。

### 技术解释

可以粗略理解为：

- 输入尺寸越大，小目标越容易保留细节
- 但尺寸越大，速度通常越慢
- 置信度阈值太低，误检会变多
- 置信度阈值太高，漏检会变多

所以这些参数不能拍脑袋定，最好让训练模型的人给一个初始建议。

## 五、确认模型到底支不支持当前系统想做的功能

### 要确认什么

我要和同学确认，当前权重究竟支持到哪一步：

- 只做框级检测
- 还是支持实例分割
- 是否适合做病害面积量化
- 是否适合做裂缝长度估算

### 为什么必须确认

因为“模型能识别”不等于“模型能支持整个系统功能”。

比如：

- `detect` 模型适合做病害存在性识别和框级展示
- `segment` 模型才更适合做掩膜展示和精细量化

如果我不把这个边界搞清楚，后面产品方案就会建立在错误预期上。

## 六、确认双模型融合时谁负责什么

### 当前推荐方案

当前最稳的方案是：

- 通用模型负责除渗水外的大部分类别
- 渗水专项模型只负责渗水

### 为什么要这么分工

因为你已经明确知道：

- 一个模型对渗水特别强
- 另一个模型对其他病害整体更稳

这种情况下，最优思路不是强迫一个模型包打天下，而是做“专家模型融合”。

### 我需要和同学确认什么

需要确认这些点：

1. 通用模型是否也输出渗水
2. 渗水专项模型是否只输出渗水
3. 通用模型在渗水类上的表现是否明显弱于专项模型
4. 专项模型是否会误把别的病害识别成渗水
5. 两个模型推荐的阈值是否不同

### 技术解释

双模型融合的核心不是简单“都跑一遍”，而是：

- 按类别分工
- 按类别融合
- 对重叠结果做去重

最简单的融合逻辑是：

```text
最终结果 =
通用模型的非渗水结果
+ 渗水专项模型的渗水结果
```

如果以后拿到更强的 PRNet 通用权重，再把“通用模型”替换掉即可。

## 七、确认是否需要保留 `PRNet-main` 运行环境

### 为什么要问这个

如果 PRNet 的权重依赖自定义模块，那系统里就不能只靠当前官方 `ultralytics` 环境。

这会影响：

- 部署方式
- 依赖管理
- 模型加载逻辑
- 后端 runner 设计

### 两种常见情况

#### 情况 A：权重可直接被官方 `ultralytics` 加载

这是最省事的情况。

那我只需要把权重路径挂进当前后端配置里即可。

#### 情况 B：权重必须依赖 `PRNet-main`

这种情况会复杂很多。

可能需要：

- 单独的 Python 环境
- 单独的模型 runner
- 或者先导出成 `onnx`

### 为什么 `onnx` 值得问

因为如果同学能给我一个导出的 `onnx`，很多自定义结构兼容问题会明显减少。

对产品工程来说，`onnx` 往往比“训练代码仓库 + 自定义 .pt”更容易稳定部署。


## 九、我接下来在工程上该怎么做

按照当前掌握的信息，后续工程动作应该分两步。

### 第一步：先把现有两个可用权重接起来

先接：

- 老通用检测模型
- 渗水专项检测模型

先做双模型融合，满足整体识别效果优先。

### 第二步：等拿到 PRNet 最终权重后再替换通用模型

如果后面同学给我真正可推理的 PRNet 权重，再判断：

- 是否能直接替换现有通用模型
- 是否要新增专用 runner
- 是否需要单独环境

### 第三步：如果目标里必须要掩膜，就继续追 `segment` 权重

如果比赛展示、量化分析、报告内容明确要求掩膜和面积，那就不能停留在 `detect` 权重。

必须进一步拿到：

- 通用 `segment` 权重
或
- 渗水专项 `segment` 权重

否则系统最多只能做到“框级识别”，做不到“真实病害区域展示”。

## 十、我现在最需要记住的结论

1. `best.pt` 只是文件名，不代表模型能力。
2. `PRNet-main` 是代码工程，不等于最终权重。
3. `detect` 模型只能出框，不能真实出掩膜。
4. 要做面积、长度、轮廓，必须优先争取 `segment` 权重。
5. 当前最现实的路线是先做“通用模型 + 渗水专项模型”的双模型融合。
6. 以后模型区分必须依赖“用途、版本、任务类型、类别列表、权重路径”，不能再靠口头记忆。
