# APP FLOW

## 作用

这份文档只回答：用户怎么走完整流程，页面应该怎么承接。

## 主线

**进入系统 -> 理解能力 -> 新建分析 -> 上传图像 -> 配置参数 -> 发起推理 -> 看结果 -> 看详情 -> 筛选对比 -> 导出 -> 回看历史**

## 流程

### 1. 进入系统

- 用户应立即知道这是桥梁病害识别系统
- 用户应立即知道输入、输出和目标
- 首页只做说明和动作入口，不堆复杂功能

### 2. 新建分析

- 这是主要入口
- 只保留必要参数
- 先告诉用户系统会输出什么

### 3. 上传与配置

- 支持拖拽或选择图像
- 校验格式与大小
- 只保留少量必要参数
- 当前模型与能力要可见

### 4. 发起推理

- 前端只发请求
- 后端决定模型版本和推理模式
- 提交后必须有明确状态反馈

### 5. 结果展示

- 原图和 `Mask` 叠加图要一屏可见
- 病害列表与图像要联动
- 结果摘要要先说结论，再说细节
- 量化字段要服务复核，不要喧宾夺主

### 6. 病害详情

- 类别
- 置信度
- 位置区域
- `Mask` 范围
- 长度、宽度、面积

### 7. 筛选与对比

- 类别筛选
- 置信度过滤
- 原图 / `Mask` / 框切换
- 模型版本对比

### 8. 导出与历史

- 导出 `JSON`
- 导出可视化结果图
- 回看历史结果
- 对照模型版本

## 视觉要求

- 深色蓝青系
- 玻璃感卡片
- 微光边框
- 低密度动效
- 结果页可以工具化，但不能像通用后台

## 异常

- 格式错误要明确拦截
- 模型不可用要返回结构化错误
- 推理失败要告诉用户能否重试

## 结论

完整 UX 不是“上传后看图”，而是“发起、理解、复核、导出、回看”的闭环。
