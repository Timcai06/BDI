# APP FLOW

## 用户体验目标

本系统的用户体验目标不是只让模型“跑出结果”，而是让用户可以完整地完成：

- 上传桥梁无人机图像
- 发起病害识别
- 理解识别结果
- 查看病害详情
- 导出结果文件
- 回看历史任务

完整 UX 的核心是让用户在尽量少的学习成本下，看懂系统能力、看懂病害结果、拿到可用输出。

## 完整 UX 主线

建议把系统完整用户体验定义为：

**进入系统 -> 选择任务方式 -> 上传图像并配置参数 -> 发起推理 -> 查看核心结果 -> 查看病害详情 -> 筛选与对比 -> 导出结果 -> 处理异常 -> 回看历史任务**

## 主流程

### 1. 进入系统

用户进入系统后，应立即理解：

- 这是桥梁病害识别系统
- 输入是无人机桥梁图像
- 输出是病害类别、Mask、置信度和结果文件
- 系统支持单张和批量识别

这一屏的目标是建立认知，而不是堆太多功能。

### 2. 选择任务方式

用户选择：

- 单张图像识别
- 批量图像识别

单张模式用于快速验证和演示。  
批量模式用于体现接近真实巡检流程的使用方式。

### 3. 上传图像

用户进入系统后上传无人机桥梁图像，并设置基础推理参数。

### 4. 配置参数

用户可以设置少量必要参数，例如：

- 置信度阈值
- 推理模式
- 是否启用高分辨率或切片推理
- 是否导出结果图

UX 原则是参数尽量少而清晰，避免系统看起来像研究脚本工具。

### 5. 发起推理

前端将请求发送给后端，后端根据配置选择模型版本与推理模式。

### 6. 获得任务反馈

用户提交任务后，系统应立即反馈当前状态，例如：

- 上传中
- 推理中
- 当前模型版本
- 当前推理模式
- 是否成功进入处理流程

如果是批量任务，还应显示进度状态。

### 7. 返回识别结果

系统返回病害类别、Mask、置信度、耗时和预留量化字段。

### 8. 结果展示

前端展示：

- 原图
- Mask 叠加图
- 病害列表
- 置信度
- 推理耗时

这一页的 UX 核心是让用户一眼看懂识别效果，而不是先面对复杂技术信息。

### 9. 查看病害详情

用户点击某一个病害后，应能查看：

- 病害类别
- 置信度
- 位置区域
- Mask 范围
- 长度、宽度、面积等量化字段

这一层用于增强结果可解释性。

### 10. 筛选与对比

用户应能：

- 按病害类别筛选
- 按置信度过滤
- 开关原图、Mask 和框
- 对比不同模型版本或不同推理策略

这一部分会显著增强系统的工具感和展示感。

### 11. 导出结果

用户可以导出结构化结果文件和可视化结果图。

建议支持：

- 单张 JSON 导出
- 单张可视化图导出
- 批量任务结果包导出

导出能力是完整 UX 的一部分，不是附属功能。

### 12. 历史结果回看

用户应能查看和重新打开历史任务，用于：

- 演示复现
- 结果对比
- 重复检查
- 模型版本对照

## 批量流程

### 1. 上传多张图像

系统创建任务并记录状态。

### 2. 批量推理

后端逐张处理并保存结果。

### 3. 查看结果

用户查看批量任务状态，并下载结果。

批量流程中，用户体验重点是：

- 看得见进度
- 看得见成功和失败数量
- 能快速拿到结果文件

## 异常流程

### 1. 图像格式错误

系统提示输入无效，阻止推理。

### 2. 模型不可用

系统返回统一错误信息，前端展示失败状态。

### 3. 推理失败

系统记录错误并允许用户重新尝试。

## 异常 UX 原则

完整 UX 不只包含成功路径，也必须包含失败路径。

每次失败都应尽量告诉用户：

- 出了什么问题
- 当前处于什么状态
- 是否可以重试
- 下一步应该做什么

## UX 总结

本项目的完整 UX 不应被理解为“上传图片然后看结果”这么简单，而应是一个从任务发起、结果理解、筛选对比、结果导出到历史复查的完整体验闭环。
