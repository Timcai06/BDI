# BACKEND STRUCTURE

## 后端目标

后端负责连接前端与算法，是系统的统一协调层。

它的任务不只是“包一层 API”，而是把模型能力组织成一个可调用、可追踪、可导出、可演示的系统能力层。

## 后端在系统中的角色

后端应承担三类桥梁职责：

- 向上承接前端交互
- 向下调用算法推理
- 横向管理结果、日志和导出

前端不应直接理解模型细节。  
算法模块也不应直接承担产品级输出和任务管理职责。  
因此后端必须成为前后端与算法之间的稳定中枢。

## 分层原则

后端内部应按职责分层，而不是把所有逻辑写进路由函数。

建议至少分为五层：

### 1. API 层

负责：

- 路由定义
- 参数接收
- 请求校验
- 状态码返回

### 2. Service 层

负责：

- 推理任务编排
- 调用模型适配层
- 调用结果处理与导出逻辑
- 组织最终返回结果

### 3. Model Adapter 层

负责：

- 加载模型
- 切换模型版本
- 切换推理后端
- 屏蔽 `.pt`、`onnx`、`engine` 差异

### 4. Postprocess 层

负责：

- 结果标准化
- Mask 处理
- 量化字段填充
- 可视化结果生成

### 5. Storage 层

负责：

- 保存原图
- 保存 JSON
- 保存叠加图
- 保存任务记录与日志

第一阶段推荐使用本地文件系统完成这些职责，不把 `Supabase` 之类的 BaaS 作为核心依赖。

更合适的策略是：

- 当前实现 `LocalFileRepository`
- 结构上预留 `ResultRepository` 抽象接口
- 未来如需接 `Supabase`，新增 `SupabaseRepository`

这样可以在不破坏现有 API 和结果协议的前提下逐步升级存储层。

## 核心职责

### 1. 请求接收

- 接收图像
- 校验参数
- 生成任务标识

请求接收层应尽早拦截非法请求，优先校验：

- 图像是否存在
- 文件格式是否支持
- 参数是否有效
- 模型版本是否可用
- 推理模式是否受支持

### 2. 推理编排

- 选择模型版本
- 选择推理模式
- 调用统一推理接口

推理编排层建议具备以下判断能力：

- 走单图还是批量流程
- 走整图还是切片推理
- 使用哪个模型版本
- 使用哪个推理后端
- 是否启用可视化和量化逻辑

这层的作用是把“产品请求”翻译成“可执行推理任务”。

### 3. 结果标准化

- 将 YOLOv8 原始输出转换为统一 JSON
- 统一字段命名和结构

这是后端中最关键的稳定层之一。

它必须保证：

- 前端永远接收到统一结果格式
- 模型内部变化不会直接冲击前端
- 新模型接入时只需要改适配逻辑，而不是全系统重写

### 4. 结果管理

- 保存 JSON
- 保存叠加图
- 保存日志

建议结果管理同时承担：

- 保存任务输入
- 保存任务输出
- 保存失败状态
- 支持历史任务查询

这样后面无论是答辩演示、错误追踪，还是模型版本对比，都会容易很多。

第一阶段建议保存的文件至少包括：

- 原始图像
- 标准化 JSON 结果
- 可视化叠加图
- 任务元信息
- 失败日志

## 建议的最小接口集合

对第一阶段产品来说，后端不需要很多接口，但必须有几个关键接口稳定存在。

### 1. `POST /predict`

用于单张图像识别。

职责：

- 接收图像
- 发起推理
- 返回统一结构化结果

### 2. `POST /predict-batch`

用于批量任务创建。

职责：

- 接收多张图像
- 创建任务
- 返回任务标识

### 3. `GET /results/{id}`

用于获取某次任务的结构化结果。

### 4. `GET /results/{id}/overlay`

用于获取某次任务的可视化叠加结果。

### 5. `GET /tasks/{id}`

用于查询批量任务进度和状态。

### 6. `GET /health`

用于检查服务是否正常、模型是否已加载。

## 推荐的存储策略

第一阶段不要为了“像产品”而过早引入 BaaS。

更稳妥的方案是：

- 图像、JSON、叠加图全部走本地文件存储
- 任务记录走本地轻量结构或文件索引
- 用统一 repository 接口隔离未来云端存储实现

只有当项目进入以下场景时，再考虑正式接入 `Supabase`：

- 多用户访问
- 云端历史记录查询
- 跨设备查看结果
- 需要持久化任务元数据和文件链接

## 建议的任务状态模型

为了支撑完整 UX，后端应显式维护任务状态，而不是只靠请求是否成功来判断。

建议至少包含：

- `pending`
- `running`
- `success`
- `failed`

如果是批量任务，还可扩展：

- `partial_success`

这样前端就能稳定展示：

- 正在处理
- 部分失败
- 已完成

## 建议的错误返回原则

后端必须返回结构化错误，而不是只返回模糊字符串。

错误信息至少应包含：

- 错误码
- 错误说明
- 可选上下文信息

例如：

```json
{
  "error": {
    "code": "INVALID_IMAGE_FORMAT",
    "message": "Only jpg, jpeg and png files are supported.",
    "details": null
  }
}
```

这样前端才能稳定处理异常 UX。

## 与前端的关系

后端对前端的核心承诺是：

- 接口路径尽量稳定
- 字段命名尽量稳定
- 成功和失败返回结构清晰
- 不暴露模型内部实现细节

前端看到的应始终是“系统结果”，而不是“模型原始输出”。

## 与算法层的关系

后端对算法层的核心约束是：

- 模型必须通过统一适配入口被调用
- 模型原始结果必须进入标准化流程
- 模型升级优先通过配置切换，而不是改 API

建议后端与算法层之间固定这种调用关系：

```text
load_runner(model_spec) -> runner
runner.predict(image, infer_spec) -> raw_result
adapt_result(raw_result, context) -> standardized_result
```

这样当前可以直接接 `YOLOv8 .pt`，后续也可以平滑接 `ONNX Runtime` 或 `TensorRT`。

这决定了你和算法同学可以真正并行协作。

## 与完整 UX 的关系

后端结构必须服务于前面定义的完整 UX，而不是单纯服务于模型调用。

具体来说，完整 UX 要求后端支持：

- 上传后的即时反馈
- 推理中的状态更新
- 统一结果展示
- 结果导出
- 历史任务回看
- 失败状态清楚可见

如果后端只会“收图 -> 跑模型 -> 返回结果”，那它支撑不了完整的系统体验。

## 第一阶段实现原则

第一阶段后端实现不必追求复杂分布式结构，但必须保证以下原则成立：

- 路由、推理编排、结果处理尽量分层
- 模型调用与业务逻辑分离
- 结果标准化独立存在
- 错误返回有统一结构
- 结果文件可追踪和可回放

## 关键原则

- API 不直接依赖某个权重文件
- 模型调用通过统一适配层完成
- 量化逻辑独立于模型调用逻辑
- 结果标准化优先于临时字段拼接
- 任务状态显式可见
- 存储实现可替换，但 API 和结果协议保持稳定

## 总结

后端的本质任务不是“提供一个推理接口”，而是把模型能力组织成一个可调用、可追踪、可导出、可演示的系统能力层。
