# FRONTEND GUIDELINES

## 前端目标

前端的首要目标不是复杂视觉，而是让评审能够快速理解系统能力和识别结果。

更具体地说，前端要完成三件事：

- 清楚表达系统是做什么的
- 直观表达算法识别结果
- 稳定承接后端统一结果协议

## 展示导向

这是一个比赛系统原型，因此前端的核心不是“做得像通用管理后台”，而是“让人快速看懂系统能力”。

前端应优先强化：

- 输入是什么
- 输出是什么
- 结果是否可信
- 系统是否具备实际应用感

因此页面设计应围绕“上传、识别、解释、导出”四件事展开。

## 信息层级原则

所有页面都应遵循同一信息层级：

### 第一层：系统能力

用于回答：

- 这是什么系统
- 解决什么问题
- 支持什么输入和输出

### 第二层：当前任务状态

用于回答：

- 当前有没有任务
- 任务是否成功提交
- 现在在上传、推理还是完成

### 第三层：核心识别结果

用于回答：

- 检出了哪些病害
- 位置在哪里
- 置信度是多少
- 效果看起来如何

### 第四层：扩展信息

用于回答：

- 病害量化结果是什么
- 导出文件在哪里
- 历史结果如何回看

## 页面重点

### 1. 上传页

- 上传图像
- 选择推理参数
- 发起任务

上传页应重点做到：

- 操作入口明显
- 参数数量可控
- 支持单图与批量两种方式
- 在上传前就告诉用户系统会输出什么

建议核心组件包括：

- 上传区域
- 参数面板
- 开始识别按钮
- 当前模型与模式说明

### 1.5 首页或说明区

如果系统有首页，建议它只承担说明职责，不要堆太多无关信息。

首页应让用户在数秒内理解：

- 系统面向桥梁病害识别
- 当前以 YOLOv8 为核心
- 支持输出病害类别、Mask、置信度和结果文件
- 支持后续扩展量化能力

### 2. 结果页

- 原图展示
- Mask 叠加展示
- 病害列表
- 置信度展示
- 推理耗时展示

结果页是整个前端的核心页面，建议采用“主画面 + 侧边信息面板”的布局。

推荐信息结构：

- 主区域：原图 / Mask 叠加图
- 侧边栏：病害列表、数量统计、推理信息、筛选控件
- 底部或详情面板：选中病害的详细信息

结果页必须优先保证：

- 原图和叠加图切换清楚
- 病害列表与图像区域联动
- 用户点击病害后能快速定位
- 关键结果一屏可见

建议核心组件包括：

- 图像查看器
- Mask 开关
- 病害列表
- 详情抽屉或侧边栏
- 导出按钮
- 推理耗时标签

### 2.5 结果详情交互

结果详情交互建议支持：

- 点击列表高亮图像中的病害
- 点击图像区域同步选中对应病害
- 展示类别、置信度、位置、量化字段
- 支持按类别和置信度筛选

这样前端不只是“展示图”，而是具备基本分析能力。

### 3. 批量任务页

- 任务状态
- 结果下载

批量任务页的核心是“管理感”和“进度感”。

建议重点展示：

- 总任务数
- 已完成数
- 失败数
- 当前状态
- 一键下载结果

### 4. 历史任务页

如果时间允许，建议加入简单的历史任务页。

它可以支持：

- 查看历史任务记录
- 打开历史识别结果
- 对比不同模型版本结果

这会明显增强系统的完整性和答辩展示力。

## UI 原则

- 信息清晰优先于炫技
- 病害类别颜色统一
- 关键结果一屏可见
- 桌面优先，同时兼顾移动端查看

补充原则如下：

- 图像区域优先级高于装饰性元素
- 结果页优先级高于首页视觉包装
- 参数设置尽量少而明确
- 状态变化要有即时反馈
- 错误信息要能被普通用户理解

## 视觉原则

前端视觉风格应偏专业、稳定、清晰，不建议走过度花哨路线。

建议：

- 以浅色或中性色界面为主
- 用有限的强调色区分病害类别
- 控制阴影、动效和装饰元素数量
- 保证图像和识别结果始终是视觉中心

比赛场景下，最重要的是“清楚”和“可信”，不是视觉炫技。

## 状态反馈原则

前端必须明确处理这些状态：

- 未上传
- 上传中
- 推理中
- 推理成功
- 推理失败
- 批量处理中
- 批量部分失败

每种状态都应有清楚反馈，而不是只靠按钮禁用或页面空白。

## 响应式原则

虽然系统主要面向桌面展示，但仍建议具备基本响应式能力。

重点保证：

- 在较窄屏幕下核心信息不丢失
- 图像区域仍可查看
- 侧边栏可折叠
- 导出和任务操作仍可触达

## 与后端协议的关系

前端必须严格建立在统一结果协议之上。

前端默认消费的核心字段应为：

- `image_id`
- `inference_ms`
- `model_version`
- `detections`
- `category`
- `confidence`
- `bbox`
- `mask`
- `metrics`

前端不应自行推断模型逻辑，也不应依赖框架内部结构。

## 前端不应承担的职责

- 不解析模型原始输出
- 不处理张量对象
- 不依赖某个具体模型内部字段

## 总结

前端的任务不是“把结果画出来”这么简单，而是把识别能力转化为用户能理解、评审能看懂、系统能稳定演示的产品体验。
