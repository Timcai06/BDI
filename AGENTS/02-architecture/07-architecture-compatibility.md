# 系统架构与兼容策略

## 总体结构

建议采用五层结构：

1. 前端交互层
2. 业务 API 层
3. 推理编排层
4. 模型适配层
5. 结果与可视化层

## 兼容策略

### 1. 模型兼容

支持：

- 原始 `YOLOv8 .pt`
- 优化后的 `YOLOv8` 变体
- 导出的 `ONNX`
- 导出的 `TensorRT`

核心原则不是“支持很多模型名字”，而是：

- 统一模型注册方式
- 统一模型调用入口
- 统一结果协议

建议至少维护如下模型配置字段：

- `model_name`
- `model_version`
- `weights_path`
- `backend`
- `imgsz`
- `device`
- `supports_masks`
- `supports_sliced_inference`

### 2. 推理兼容

支持：

- 整图推理
- 切片推理

建议推理模式至少统一为：

- `direct`
- `sliced`

这样后续算法同学即使优化高分辨率策略，也不会冲击前后端交互方式。

### 3. 结果兼容

统一输出：

- `category`
- `confidence`
- `bbox`
- `mask`
- `metrics`

建议结果协议至少固定这些顶层字段：

- `schema_version`
- `image_id`
- `inference_ms`
- `model_name`
- `model_version`
- `backend`
- `inference_mode`
- `detections`
- `artifacts`

这样前端只认标准结构，不认 Ultralytics 原始 `Results` 对象。

## YOLOv8 接入方式

第一阶段建议采用 `Python` 内嵌推理方式：

- `FastAPI` 直接调用 `Ultralytics YOLOv8-seg`
- 模型启动时或首次请求时加载
- 推理后返回原始结果对象
- 结果适配层统一转换成系统协议

这一方案的优点是：

- 联调最快
- 跨语言成本最低
- 最适合比赛原型
- 后续仍可演进到 `ONNX` 或 `TensorRT`

## 优化后模型的接入方式

当算法同学完成一轮优化后，系统不应把它当作“临时替换一个模型文件”，而应把它当作“注册一个新的模型版本”。

这意味着：

- 前端不需要重写
- API 路径不需要变化
- 你主要调整模型注册配置和结果适配层

推荐思路是：

- 新模型进入模型注册表
- 新模型通过统一推理入口调用
- 新模型输出通过统一适配层转成标准协议

## 算法同学每次交付新模型时应提供的内容

建议每次交付至少包含：

- `model_name`
- `model_version`
- `weights_path`
- `backend`
- `imgsz`
- 类别映射说明
- 是否支持 `mask`
- 是否支持切片推理
- 是否有特殊后处理
- 一组测试图片与参考结果

这样做可以把“模型优化”变成受控版本升级，而不是一次高风险的临时替换。

## 你接入优化后模型时需要做的事情

在理想结构下，你通常只需要做两件事：

### 1. 增加模型注册信息

把新模型作为新版本加入注册表。

### 2. 检查适配层输出

如果新模型原始结果结构与当前兼容，则前后端几乎不需要改动。  
如果 `mask`、类别映射或后处理逻辑发生变化，则只在适配层修正，不改前端协议。

## 推荐的接入验证流程

每次接入优化后的模型版本时，建议都执行同一套流程：

1. 检查模型是否可加载
2. 跑通单张图推理
3. 检查结果字段是否符合统一协议
4. 验证前端结果页是否正常显示
5. 验证 JSON 和叠加图是否正常导出
6. 验证批量任务与历史任务是否正常

这套流程的目的，是把关注点放在“系统兼容性”上，而不是重新研究模型细节。

## 后续优化边界

算法同学后续可以优化：

- 训练数据与增强
- 权重版本
- 小裂缝和复杂场景表现
- 推理后端
- 后处理逻辑

但必须保持以下边界：

- 不随意改最终输出字段
- 不让前端依赖类别 ID 细节
- 不让前端解析原始 `Results`
- 如果 `mask` 表达变化，在适配层统一解决
- 如果类别映射变化，在配置层解决
- 模型升级视为配置变化，而不是接口变化
- 每次升级都应保留可回归测试的样例输入与参考输出

## 设计原则

- 前端只依赖统一结果
- 后端只调用统一推理接口
- 模型可以升级，协议尽量不变
- 存储可以从本地文件升级到 BaaS，但业务层不重构
