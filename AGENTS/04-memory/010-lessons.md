# LESSONS

## 作用

这份文档只保留能直接影响后续编码的经验。

## 经验

### 1. 先定方向，再定实现

方向不收敛，后面一定返工。

### 2. 不要绑定单一权重文件

核心是可兼容的推理框架，不是某个权重本身。

### 3. 比赛更看重系统化表达

不仅要有指标，还要有结果展示、导出、解释和演示。

### 4. 前后端的价值在于把算法变成作品

必须把展示闭环做完整。

### 5. 算法升级不能破坏产品接口

升级应落在模型层和适配层，不应冲击前端协议。

### 6. Mock 闭环能显著降低联调风险

先跑通统一 schema、错误结构和状态流，再接真实模型。

### 7. 真实模型依赖应与基础开发依赖解耦

重型依赖不要绑死基础开发环境。

### 8. Python 版本会直接影响模型接入

真实模型环境按 `Python 3.9` 到 `Python 3.12` 控制，新增代码也要按这个标准写。

### 9. 代码已落位不等于功能已完成

必须区分：

- 代码存在
- mock 验证通过
- 真实权重验证通过

### 10. 真实模型环境要锁定依赖组合

装了包不等于能用，关键版本要固化。

### 11. 沙箱报错不一定等于项目报错

底层运行时问题要区分沙箱限制和真实故障。

### 12. 阶段完成后必须立即关账

状态一旦落地，就要同步入口、计划和记忆文档。

### 13. 先读 AGENTS，再改代码

执行规范和阶段状态都在 `AGENTS/`。

### 14. Phase 2 完成不等于展示完成

MVP 闭环只是基础，展示增强属于后续阶段。

### 15. 历史回看应先复用统一结果文件

先用本地文件跑通回看，再考虑数据库。

### 16. 导出能力优先按介质拆分

`JSON` 和 overlay 分开处理最稳。

### 17. 用户需要的是记录工作台，不是上传表单

历史列表、回看、比较、继续处理，优先级高于表单细节。

### 18. 首页必须传达系统里已经有什么

没有历史和入口，用户会把系统理解成一次性工具。

### 19. 新建分析更适合作为动作

它应是高优先级动作入口，不是长期导航页面。

### 20. 历史页一旦记录增多，就必须提供检索

至少要有关键词搜索和病害类别筛选。

### 21. 历史页必须具备管理能力

要有概览、搜索、批量操作、分页。

### 22. 卡片布局的信息层级必须清晰

图片、关键信息、次要信息、操作按钮要分层处理。

### 23. 双模型融合要提前声明融合边界

框级融合与掩膜级融合是两回事。当前模型权重是 `detect` 类型，只能做框级融合，不能做像素级掩膜输出。这个限制必须在文档和交互中明确说明，避免用户误解。

### 24. 数据库落位后文档需同步更新 schema 变更

新增表、字段、迁移脚本后，`AGENTS/` 中的后端结构文档必须同步更新，否则后续开发会基于过时的文档做决策。

### 25. 前端页面壳统一应在页面增多前完成

当页面数量超过 5 个时，头部不一致、布局不统一的问题会迅速放大。共享页面壳（`PageShell`）和统一头部语义应在页面骨架阶段就收口。

### 26. 桥梁资产作为一级入口后，批次流程的 UX 重心随之转移

桥梁不再是被动关联字段，而是批次创建、查看、统计的第一级导航。这个变化影响批次中心、运营总览、病害检索等多个页面的信息架构。

### 27. Phase 阶段文档的"尚未具备"清单在实现后必须立即关账

`plan/phaseX.md` 中的"当前尚未具备"和"下一步"在功能落地后必须同步更新，否则会给后续阅读者造成"功能未完成"的错误判断。

### 28. ops 工作台从展示到处置的改造需要共享布局先行

处置型界面要求高风险优先展示、批量操作入口可见、时效信息突出。这些需求必须在共享布局层面解决，而不是每个页面各自实现。

### 29. 桥梁资产作为独立路由后，导航结构需要重新排序

侧边栏中"桥梁资产"应排在"运营总览"之后、"批次工作台"之前，体现"先看资产、再做巡检"的业务逻辑。

### 30. 全量中文化不是简单的文字替换

英文标签替换为中文时，需要同时调整字号、字重和间距以保持视觉层级。按钮文字替换为图标时，必须补充 `title` 属性保证可访问性。

### 31. Bento 式布局需要动画连接元素

分步导航的两块面板之间用箭头或进度指示器连接，能显著降低用户的认知负担。`framer-motion` 的 `layoutId` 和路径动画是关键。

### 32. demo 降级策略能显著降低演示风险

当 `API_BASE_URL` 未配置时，自动回退到演示产物路径，确保 UI 在无后端环境下仍可完整展示。这对比赛评审和团队演示至关重要。

### 33. 增强处理从自动到可选的 UX 权衡

将 `enhancement_mode` 默认值从 `auto` 改为 `always`，同时新增事后增强接口，让用户既可以依赖默认增强，也可以对已有结果按需触发。这种"默认开 + 手动兜底"的模式比"自动判断是否需要"更可预测。
