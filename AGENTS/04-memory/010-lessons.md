# LESSONS

## 作用

这份文档只保留能直接影响后续编码的经验。

## 经验

### 1. 先定方向，再定实现

方向不收敛，后面一定返工。

### 2. 不要绑定单一权重文件

核心是可兼容的推理框架，不是某个权重本身。

### 3. 比赛更看重系统化表达

不仅要有指标，还要有结果展示、导出、解释和演示。

### 4. 前后端的价值在于把算法变成作品

必须把展示闭环做完整。

### 5. 算法升级不能破坏产品接口

升级应落在模型层和适配层，不应冲击前端协议。

### 6. Mock 闭环能显著降低联调风险

先跑通统一 schema、错误结构和状态流，再接真实模型。

### 7. 真实模型依赖应与基础开发依赖解耦

重型依赖不要绑死基础开发环境。

### 8. Python 版本会直接影响模型接入

真实模型环境按 `Python 3.9` 到 `Python 3.12` 控制，新增代码也要按这个标准写。

### 9. 代码已落位不等于功能已完成

必须区分：

- 代码存在
- mock 验证通过
- 真实权重验证通过

### 10. 真实模型环境要锁定依赖组合

装了包不等于能用，关键版本要固化。

### 11. 沙箱报错不一定等于项目报错

底层运行时问题要区分沙箱限制和真实故障。

### 12. 阶段完成后必须立即关账

状态一旦落地，就要同步入口、计划和记忆文档。

### 13. 先读 AGENTS，再改代码

执行规范和阶段状态都在 `AGENTS/`。

### 14. Phase 2 完成不等于展示完成

MVP 闭环只是基础，展示增强属于后续阶段。

### 15. 历史回看应先复用统一结果文件

先用本地文件跑通回看，再考虑数据库。

### 16. 导出能力优先按介质拆分

`JSON` 和 overlay 分开处理最稳。

### 17. 用户需要的是记录工作台，不是上传表单

历史列表、回看、比较、继续处理，优先级高于表单细节。

### 18. 首页必须传达系统里已经有什么

没有历史和入口，用户会把系统理解成一次性工具。

### 19. 新建分析更适合作为动作

它应是高优先级动作入口，不是长期导航页面。

### 20. 历史页一旦记录增多，就必须提供检索

至少要有关键词搜索和病害类别筛选。

### 21. 历史页必须具备管理能力

要有概览、搜索、批量操作、分页。

### 22. 卡片布局的信息层级必须清晰

图片、关键信息、次要信息、操作按钮要分层处理。
