# LESSONS

## 当前阶段经验

### 1. 先定方向，再定实现

如果项目方向没有先收敛，后面的前后端和算法对接会不断返工。

### 2. 系统不能绑定单一权重文件

项目的核心不是某一个 YOLOv8 权重，而是一个可兼容 YOLOv8 及其优化变体的推理框架。

### 3. 比赛更看重系统化表达

赛题不仅看模型指标，也看“核心算法系统”和“实际应用效果展示”。

### 4. 前后端的价值在于把算法变成作品

真正的差异化不只是把模型跑通，而是把结果展示、导出、解释和演示做完整。

### 5. 算法升级不能破坏产品接口

算法优化应被视为模型层和适配层的变化，而不是前后端协议的变化。只要输出协议稳定，系统就能持续演进而不频繁返工。

### 6. Mock 闭环能显著降低联调风险

在真实模型接入前，先把统一 schema、错误结构和前后端状态流跑通，可以把“协议问题”和“模型问题”拆开处理，联调效率会高很多。

### 7. 真实模型依赖应与基础开发依赖解耦

像 `ultralytics` 这类重型依赖，不适合强绑在所有基础环境里。拆成可选安装后，前后端骨架开发、测试和协议联调就不会被模型依赖阻塞。

### 8. Python 版本会直接影响模型接入节奏

当前机器的 `Python 3.14` 可以支撑基础 FastAPI 开发，但 `ultralytics` 依赖的 `torch` 暂时无法直接安装，因此真实模型接入建议单独使用 `Python 3.9` 到 `Python 3.12` 的虚拟环境。

### 9. “代码已落位” 不能等同于 “功能已完成”

像真实 runner 工厂、adapter 和前端统一 schema 这类能力，即使代码已经写好，也必须区分：

- 代码已存在
- mock 路径已验证
- 真实权重场景已验证

只有最后一层也通过，才能把该项记为真正完成。否则文档会比实际进度超前。

### 10. 真实模型环境需要锁定依赖组合

这次验证里，`.venv-yolo` 虽然已经安装了 `ultralytics`，但 `torch 2.2.2` 与 `NumPy 2.0.2` 组合会触发导入兼容问题。对于真实模型联调，除了记录“装了什么包”，还需要把关键版本约束固化到依赖文件里。

### 11. 沙箱内的模型报错不一定等于项目报错

这次 `torch` 在沙箱内触发了 OpenMP 共享内存错误，但在沙箱外验证后，`ultralytics + torch` 可以正常导入。涉及底层运行时的模型依赖时，需要把“沙箱限制”和“项目真实故障”区分开。
