# Phase 5 执行说明

## 目标

本阶段的目标是将系统从“单图识别演示”升级为“可承接真实无人机巡检业务流程的企业化系统原型”。

核心不是继续堆叠展示项，而是先把后端主语从“单张预测结果”改成“批次任务系统”，让系统能够支撑批量图片接入、异步处理、结果追踪、病害检索、人工复核和预警闭环。

## 开工前提

进入本阶段前，默认以下事项已经完成并验收：

- `Phase 4` 多模型骨架与双模型融合路径已建立。
- 前后端统一结果协议已经稳定，双模型识别可通过统一 runner / adapter 输出标准化结果。
- 单图识别、历史结果查看、结果导出等演示链路已打通。

## 范围

本阶段优先承诺落地以下能力：

- 批次为核心的后端领域模型与数据库结构
- 图片批量接入与异步任务编排
- `batch / batch_item / task / result / detection` 一体化追踪
- 病害检索、聚合统计与人工复核
- 基于规则的预警事件预留与初版落地
- 面向企业流程的 Dashboard 骨架

## 实施顺序

1. 先重构后端数据模型与 API，把系统主语切换到 `batch workflow`。
2. 再接本地异步 worker，把图片接入、任务执行、结果入库跑通。
3. 再建设企业化 dashboard，围绕批次、详情、筛选、复核来组织前端。
4. 最后补预警规则、统计接口和后续扩展点。

## 关键约束

- 不再以 `POST /predict -> 单图返回结果` 作为正式业务主入口。
- 元数据、检测记录、复核记录、预警事件必须入数据库，不能继续只依赖文件系统 JSON。
- 结果产物文件仍可保留本地存储，但结构化查询必须基于数据库。
- 第一版先解决图片批次与异步任务，不先做视频解析、WebSocket 和复杂审批流。
- 本地开发优先兼容 `SQLite`，正式系统目标数据库定为 `PostgreSQL`。

## 完成定义

满足以下条件视为 `Phase 5` 完成：

- 系统可以创建桥梁巡检批次并持续接收多张图片。
- 上传图片后可自动入队并由异步任务完成识别。
- 前端可查看批次进度、单张详情、检测结果与人工复核状态。
- 后端可以按病害类别、置信度、面积等字段检索检测记录。
- 系统可生成并追踪初版预警事件。
