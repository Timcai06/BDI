# Phase 4 执行说明

## 目标

本阶段的目标是让系统具备承接后续算法优化模型的能力，同时补齐批量处理和高分辨率推理能力。

核心不是简单增加功能数量，而是把系统从“当前可跑通 `YOLOv8`”升级为“算法可插拔、协议稳定的推理框架”。

## 开工前提

进入本阶段前，默认以下事项已经完成并验收：

- `Phase 2` 真实单图闭环已完成
- `Phase 3` 结果展示、导出、历史回看等能力已完成
- 前端可稳定消费统一结果协议
- 真实 `YOLOv8-seg` 单图链路已经跑通

## 范围

本阶段优先承诺以下能力：

- 模型版本注册机制
- 统一 runner / adapter 骨架
- 稳定统一结果协议
- 支持切换不同 `YOLOv8` 版本
- 支持 `direct` 和 `sliced` 推理模式
- 批量任务创建与查询
- 优化模型接入流程文档化
- 历史任务与版本区分增强

## 实施顺序

1. 先接入第二个真实模型版本
2. 再验证“换权重 + 少量配置”的真实流程
3. 再推进更清晰的模型可用性提示
4. 然后推进批量任务和 `sliced` 推理
5. 最后增强更细粒度的差异表达

## 关键约束

- 前端继续只消费统一结果协议，不依赖模型私有字段
- 模型升级应被视为配置变化或适配层变化，而不是接口变化
- 批量与切片能力应建立在统一推理骨架和真实模型验证基础上
- 历史记录应继续区分模型版本、推理模式和产物来源

## 当前状态

按当前仓库真实代码，以下 Phase 4 能力已经落地：

- `ModelSpec` / `ModelRegistry` / `RunnerManager` 骨架
- `model_name` 与 `model_version` 的基础分离
- `/predict` 按 `model_version` 选择 runner
- `GET /models` 模型列表接口
- 前端模型版本选择
- 同图双模型对比
- 历史记录再次模型对比
- 图像级并排对比
- 病害类别级差异对比
- `inference_mode` 参数 (`direct` / `sliced`)
- `supports_sliced_inference` 字段
- 模型可用性 (`is_available`) 检查
- `sliced` 推理模式预留接口
- 模型系统关系说明文档
- 新模型接入操作手册

因此当前状态更准确地说是：

- `Phase 4` 已完成第一轮多模型骨架建设
- 当前重点是接入第二个真实模型版本并验证切换路径

## 完成定义

满足以下条件视为 `Phase 4` 完成：

- 系统可以接入至少两个模型版本
- 切换模型时前端无需重构
- 同类优化模型可以通过替换权重和少量配置完成切换
- 差异更大的新算法可以通过适配层接入，而不破坏主流程
- 批量任务可创建、查询、导出
- 优化后的模型可按统一流程接入
