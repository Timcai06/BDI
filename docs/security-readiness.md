# Security Readiness

## 本轮范围

当前环境按内部/演示使用处理，本轮不直接上线 JWT 或权限系统。

这份文档只锁定下一轮安全改造的边界，避免后续重复讨论。

## 1. 认证入口

建议认证入口放在 FastAPI API 层，而不是散落在 service 层：

- 统一接入点：`backend/app/api/`
- 方式：
  - `Depends(get_current_user)` 或等价认证依赖
  - 公共鉴权中间件只做最基础拦截，不承载业务授权

## 2. 需要保护的路由

下一轮至少保护：

- `/api/v1/batches/*`
- `/api/v1/bridges/*`
- `/api/v1/tasks/*`
- `/api/v1/detections`
- `/api/v1/reviews`
- `/api/v1/alerts`
- `/api/v1/ops/*`
- `/results/*` 下载接口

可以保留匿名的：

- `/health`

## 3. 前端 token 存储策略

推荐优先级：

1. HttpOnly Cookie
2. 如必须前端可读，再考虑内存态 + 刷新机制

不建议：

- 长期令牌直接落 `localStorage`

## 4. 最低授权模型

至少区分：

- 运营查看
- 批次操作
- 资产管理
- 系统配置
- 审计查看

不要让“能看页面”直接等于“能删桥梁/删批次/重跑任务”。

## 5. 审计日志最低要求

下一轮落安全时，以下动作要有审计记录：

- 创建/删除桥梁
- 创建/删除批次
- 重试任务
- 触发增强补算
- 修改告警规则
- 修改系统设置

审计最少字段：

- 操作人
- 动作类型
- 目标对象类型 / ID
- 时间
- 请求来源
- 结果（成功 / 失败）

## 6. 限流位置

建议分两层：

- 反向代理层：统一 IP / 路径级限流
- 应用层：对高成本接口做用户级限流

高成本接口包括：

- 预测/重跑
- 批量上传
- 增强补算
- 结果导出

## 7. 上线前检查清单

下一轮安全改造开始前，先确认：

1. 前端登录态存储方案
2. 用户与角色模型
3. 需要开放给匿名访问的最小路由集合
4. 审计日志落库位置
5. 限流放在哪一层优先实现
