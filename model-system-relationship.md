# 模型结构、权重与系统关系说明

更新时间：2026-03-14

## 这份文档解决什么问题

这份文档用于回答下面几个核心问题：

1. 什么是深度学习模型
2. 什么是模型权重
3. 为什么一个 `.pt` 文件可以接入当前项目
4. 当前系统里，模型、权重、runner、统一协议分别扮演什么角色

本文偏概念解释，但尽量贴近当前仓库实现。

## 一句话理解

你现在做的不是“直接运行一个 `.pt` 文件”，而是：

**让系统通过特定的推理引擎去加载某个模型权重文件，再把推理结果转换成统一协议，交给前端展示。**

## 一、什么是深度学习模型

可以先把深度学习模型理解成两部分：

1. 模型结构
2. 模型权重

### 1. 模型结构

模型结构决定：

- 输入是什么
- 中间有多少层
- 每层如何计算
- 输出是什么

例如：

- 这是一个检测模型还是分割模型
- 它输出框、类别，还是输出 mask
- 它内部是怎样的骨干网络和头部结构

如果类比一下，模型结构更像：

- 电路图
- 机器的装配方式
- 神经网络的“骨架”

### 2. 模型权重

模型权重是训练后学到的参数。

这些参数决定：

- 模型会关注图像里的哪些特征
- 它如何区分裂缝、剥落、泛碱等病害
- 它画框和分割 mask 的能力如何

如果类比一下，模型权重更像：

- 一个训练完成后的“大脑记忆”
- 一台机器调好的内部参数
- 同一套结构下，不同训练结果的“经验”

## 二、为什么 `.pt` 文件能接进项目

## 先说结论

不是因为 `.pt` 文件自己“会运行”，而是因为你的项目里已经有代码知道怎么解释它。

当前项目的真实推理路径里，关键不是 `.pt` 文件本身，而是：

1. 你使用的是 `Ultralytics YOLOv8`
2. 你的后端里有 `UltralyticsRunner`
3. `UltralyticsRunner` 知道如何加载 `.pt`
4. 系统再把推理结果转成统一格式

也就是说：

**`.pt` 文件只是权重载体，真正让它可用的是对应的推理框架和接入代码。**

## 当前项目里的实际链路

当前项目里，真实模型推理大致是下面这条链：

```text
前端上传图片
  ->
FastAPI 接收请求
  ->
PredictService 组织推理任务
  ->
RunnerManager 选择模型版本
  ->
UltralyticsRunner 加载并调用 YOLOv8
  ->
YOLOv8 使用 .pt 权重执行推理
  ->
RawPrediction
  ->
PredictResponse（统一协议）
  ->
前端结果页 / 导出 / 历史记录
```

## 和当前代码对应的关系

当前仓库中，这条链主要对应：

- 模型配置与注册
  - `backend/app/core/config.py`
  - `backend/app/adapters/registry.py`

- 模型选择与 runner 管理
  - `backend/app/adapters/manager.py`
  - `backend/app/adapters/factory.py`

- 当前真实 YOLOv8 接入
  - `backend/app/adapters/ultralytics_runner.py`

- mock 兜底
  - `backend/app/adapters/mock_runner.py`

- 统一结果协议
  - `backend/app/models/schemas.py`
  - `backend/app/services/predict_service.py`

## 三、为什么现在会看到 `mock-v1`

`mock-v1` 不是说系统没有 `YOLOv8`，而是：

- `model_name` 表示模型体系，例如 `yolov8-seg`
- `model_version` 表示具体版本，例如 `v1-real`、`v2-real`、`mock-v1`

所以：

- `yolov8-seg / v1-real`
  - 表示当前是 YOLOv8 真实模型版本

- `yolov8-seg / mock-v1`
  - 表示当前系统在用 mock runner 兜底联调

mock 的作用不是替代真实模型，而是：

- 前后端联调
- UI 演示
- 在真实模型环境没配好时保持系统可运行

## 五、什么时候只换权重就够

这个问题很重要。

## 情况一：只换权重就够

通常发生在下面这些场景：

1. 还是 `YOLOv8-seg`
2. 只是训练得更好
3. 输出结构和当前基本兼容
4. 仍然走当前 `UltralyticsRunner`

这时通常只需要：

- 新增一个模型版本配置
- 指向新的权重路径
- 通过模型注册表接进系统

这种情况是你当前架构最擅长支持的。

## 情况二：只换权重不够

如果你换的是：

- 完全不同结构的模型
- 完全不同推理框架
- 输出格式差异很大的模型

那就不能只靠换 `.pt` 文件。

这时还需要：

- 新增对应 runner
- 在 factory/manager 里注册
- 把输出适配成统一 `RawPrediction`

所以更准确的说法不是：

“未来任何模型都只换一个 `.pt` 就能接入”

而是：

**同类模型优先通过换权重接入，不同模型通过 runner + 适配层接入。**

## 六、为什么系统要做统一协议

因为如果前端直接读模型原始输出，问题会很多：

- 换模型前端就得重写
- 换后端推理框架前端也会被影响
- 导出、历史记录和对比功能都容易返工

所以你现在系统里最值钱的，不只是能跑 `YOLOv8`，而是：

**前端只依赖统一 schema，模型细节被隔离在 runner 和适配层里。**

这也是为什么你的项目后续能扩到：

- 多模型版本切换
- 模型对比
- batch
- sliced 推理

## 七、把这件事讲给别人时可以怎么说

你可以这样描述你当前系统：

> 当前系统以 `YOLOv8-seg` 作为第一代真实推理引擎，通过统一的模型注册、runner 管理和结果协议，把权重文件加载、模型推理和产品展示解耦开来。  
> 因此同类优化模型可以低成本替换权重接入，不同模型也可以通过新增 runner 和适配层接入，而不破坏前端和产品主流程。

## 八、最后一句最重要的话

**`.pt` 文件不是“模型本身”，而是训练后的参数载体；真正让它在项目里可用的是“模型结构 + 推理框架 + runner 接入代码 + 统一结果协议”。**
